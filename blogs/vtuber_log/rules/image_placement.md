# vtuber_log 画像選定・配置ルール（唯一の正）

記事への画像の選定・配置・alt/タイトル生成の基準。`/vtuber-article` スキルと手動作業の両方で適用する。挿入形式の完成形リファレンスは `drafts/azusa-honami-profile.txt`。

---

## 1. 画像の置き場と命名

- 記事で使う画像は `assets/<記事スラッグ>/` に入れる（記事ごとに1フォルダ）
- **assetsは使い捨ての作業場**：候補画像を多めに入れてよく、全部使う必要はない（最適なものだけ選定される）。WordPressへ貼り付けて画像表示を確認した後は、フォルダごと削除してよい（使った画像はWP側に実体があるため）。**注意：未使用画像はWPに上がっていないので削除で完全に消える**。元データが他にない画像は削除前に退避する
- フォルダの中身はgit管理対象外（履歴に画像バイナリを残さない）。記録として残るのはWP上の画像とドラフト内のURLのみ
- **ファイル名はASCIIのみ・記事スラッグ入りの一意な名前**にする（例：`deco-be22-18.webp`）。日本語名はアップロード不可（`wp_upload.ps1`が拒否する）。同名衝突するとWordPress側でリネームされURLがズレるため、汎用名（`image1.webp`等）は使わない
- 形式はwebp推奨（jpg/png/gifも可）

## 2. 挿入形式（deco-be22踏襲・変更不可）

通常の本文挿入は必ずネイティブ`wp:image`ブロック：

```html
<!-- wp:image -->
<figure class="wp-block-image size-large"><img src="（uploaded_urls.jsonの実URL）" alt="（具体的な日本語説明）" width="800"/></figure>
<!-- /wp:image -->
```

- `src`は必ず `wp_upload.ps1` が返した実URL（`uploaded_urls.json`）を使う。**URLの手組み・推測は禁止**（アップロード時にリネームされている可能性があるため）
- 縦横比を保った縮小が必要な場合のみ `is-resized`＋`height`を追加
- Q&Aカード等のデザインパーツ内に埋める場合のみ `design_components.md` 9番の`wp:html`内`<img>`形式
- `wp:image`ブロックに`style`属性やtitle属性を足さない（検証エラーの実績あり：`wordpress_output_rules.md` §4）

## 3. 配置の基準（SEOルール§2「関連テキストの近く」の具体化）

- **その画像が説明している内容を扱うセクション内**に置く。見出し直後か、言及した段落の直後
- 冒頭（アイキャッチ相当）：製品全体・パッケージなど記事の主題が一目でわかる1枚
- 外観・設置の説明 → 外観写真、実測の説明 → 測定結果のスクリーンショット、のように本文と画像の対応を1対1にする
- **枚数の目安：H2セクションごとに0〜2枚・記事全体で5〜12枚**。装飾目的で無関係な画像を挟まない
- 同じ画像を記事内で2回使わない

## 4. alt・タイトルの書き方

- **alt**：「製品名＋何の写真か＋（必要なら）注目ポイント」を具体的に書く（例：`TP-Link Deco BE22 本体背面のポート・端子部分の写真`）。空alt・絵文字のみ・キーワード羅列は禁止（QAで検出）
- **タイトル**：メディアライブラリ用の短い名前（例：`Deco BE22 背面ポート`）。本文には埋め込まない（既存仕様に`title`属性は存在しない）。`wp_upload.ps1`がmanifest.mdからメディアライブラリへ自動設定する

## 5. manifest.md（画像解析キャッシュ）の仕様

`assets/<スラッグ>/manifest.md` に以下のパイプテーブルで記録する（先頭3列は`wp_upload.ps1`のパーサーが読むため列順固定）：

```markdown
| file | alt | title | description | 配置先 |
|---|---|---|---|---|
| deco-be22-18.webp | TP-Link Deco BE22 本体とパッケージの全体写真 | Deco BE22 パッケージ | 白い本体2台と外箱 | 冒頭 |
```

- 画像を`Read`で解析して生成する。再実行時は既存manifestを再利用（画像追加分のみ解析）
- 記事に使わなかった画像は配置先を`未使用`と記録し、完了報告で知らせる

## 6. アップロードフロー（B案・REST API）

1. manifest.md完成後、`tools\wp_upload.ps1 -Slug <スラッグ> -DryRun` で対象を確認
2. 本実行でメディアライブラリへアップロード＋alt/タイトル自動設定＋`uploaded_urls.json`生成
3. ドラフトの`src`には`uploaded_urls.json`のURLを埋め込む
4. 認証は `tools\wp_auth.local.json`（gitignore済み・repoにコミットしない）。未作成ならスクリプトが作成手順を案内する
5. **初回セットアップ時のみ**：`tools\wp_connect_test.ps1` で疎通確認する（認証→画像UP→下書き投稿→cleanupの4段階。詳細はスクリプト冒頭のUsage参照）
