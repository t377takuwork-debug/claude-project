# vtuber_log WordPress貼り付け用出力ルール（唯一の正）

記事ドラフトをWordPressへそのままコピペできる形式で生成する際の技術ルール。cf_room の `deco-be22-review_revised_v2.txt` のリライト作業で実際に発生した不具合をもとに作成した、再発防止のためのルール集。

---

## 1. ファイル形式

- 拡張子は `.txt`（`.html`にしない。IDE/OSがプレビュー表示しようとして紛らわしくなるため）
- 保存先は各記事の `drafts/` フォルダ

## 2. Gutenbergブロックコメントの基本

全要素を `<!-- wp:ブロック名 -->` 〜 `<!-- /wp:ブロック名 -->` で明示的に囲む。
- 段落：`wp:paragraph`
- 見出し：`wp:heading`（h2はデフォルト。h3・h4等それ以外のレベルは `{"level":3}` のように属性を付与）
- 画像：`wp:image` ＋ `<figure class="wp-block-image size-large">` 構造
- 表：`wp:table` ＋ `<figure class="wp-block-table"><table>...</table></figure>` 構造
- カスタムデザイン（色付きボックス・グリッド・チップ等）：`wp:html`

### 貼り付け手順
1. WordPress編集画面右上メニューから「コードエディター」に切り替える
2. ドラフトの内容を全文貼り付け
3. 「ビジュアルエディター」に戻す

**ビジュアルエディターへ直接貼り付けるとブロックコメントが解釈されず、文字列としてそのまま表示される。** 必ずコードエディター経由にする。

## 3. 絶対にやってはいけないこと（実際に起きた重大バグ）

### 3-1. 説明用コメントの中に `<!-- wp:xxx -->` という文字列を書かない
HTMLコメントは最初の `-->` で閉じる。自分用の注釈コメント内で「`<!-- wp:paragraph -->` のように書く」と例示すると、その埋め込み自体が本来の閉じ位置より早くコメントを終了させてしまい、以降のブロック解析が全てずれる（実際に発生し、記事全体のブロックコメントが可視文字として表示される致命的バグになった）。
→ **ドラフトファイル本体には、貼り付け対象の実コンテンツ以外の説明・注釈を一切含めない。** 補足説明はチャット側の返信でのみ行う。

### 3-2. `wp:html` ブロックの中に別の `<!-- wp:xxx -->` を入れ子にしない
Gutenbergのパーサーはブロックコメントをフラットに走査するため、`wp:html` の内容として書いたつもりの `<!-- wp:paragraph -->` 等が、意図通り「ただの文字列」として扱われる保証がない。ユーザーから貼られたコード片に既存の `wp:paragraph` が含まれている場合は、それを自分の `wp:html` で二重に囲まず、そのまま独立したブロックとして配置する。

## 4. ネイティブブロック（画像・表など）特有の注意

- `core/image` ブロックは、ブロックコメントのJSON属性（例：`{"align":"center"}`）と実際のHTML（`class="wp-block-image aligncenter"`）が対応している必要がある。**属性にない `style` を figure タグに追加すると「このブロックには、想定されていないか無効なコンテンツが含まれています」という検証エラーになる**（実際に発生）。
- 画像を中央寄せしたい等の見た目調整は、ネイティブ `wp:image` ブロックの属性を無理に拡張せず、`wp:html` のカスタムデザインとして組む方が安全。
- `wp:html`（カスタムHTMLブロック）は保存内容をそのまま出力するだけで検証が行われないため、任意のインラインstyle・入れ子構造を入れても崩れない。

## 5. スタイル（インラインCSS）の実装ルール

### 5-1. フォントサイズ・行間はテーマCSSに上書きされることがある
vtuber-verse.com のテーマは `@media (max-width:599px)` 内で `.post .entry-content ul li` や `.post .entry-content p` 等、かなり詳細なセレクタに `!important` 相当の強制指定を持っている。**指定した子要素（span等）ではなく、親のliやul、あるいはpタグ自体が上書きされるため、狙った見た目にならないことがある。**

対策：
- font-size・line-height・colorなどを固定したい要素は、**その要素自身（実際にテキストを持つ最も内側のタグ）に直接 `!important` 付きで指定する**。親要素だけに指定して継承に頼らない。
- スマホ側だけ小さくしたい等の要望に `clamp(min, Nvw, max)` を使う場合、**実機幅（375px・390px・428px等の主要スマホ幅）で計算値を検算すること**。vw係数が大きすぎると、どのスマホ幅でもmax値に張り付いてしまい、意図した縮小効果が出ない（実際に `3.5vw` が常に14px上限に到達してしまうバグが発生）。用途によっては vw を使わず固定pxで統一する方が安全で確実。
- スマートクォート（U+201C/201D/2018/2019）を混入させない。HTML属性が壊れる原因になる。

### 5-2. デザインバリエーション
同じ見た目（ピル型チップ等）を記事内で3回以上使い回すと単調になる。使い分けの一覧は `design_components.md` 末尾「使い分けの原則」を唯一の正とする（ここには重複記載しない）。

## 6. 検証手順（編集の都度必ず実施）

専用スクリプトを実行する。手打ちのワンライナーは使わない（過去に検証コード自体の文字化けで誤検査が起きたため）。

```powershell
powershell -ExecutionPolicy Bypass -File "C:\Users\PC_User\claude project\blogs\vtuber_log\tools\qa_draft.ps1" "対象ドラフトのパス"
```

チェック内容：ブロック開閉数の一致／コメント総数の一致／スマートクォート混入／JSON-LD構文／公開前プレースホルダー残置（`href="#"`・「要修正」・商品リンク仮置き）／空alt属性／`wp:html`内へのブロックコメント入れ子（3-2）／内部リンクの実在（`internal_links.md`照合）／`MANUAL_JSONLD`目印の有無（7）／JSON-LDと本文の同期（headline＝タイトル行・FAQ数＝本文Q&A数）／内部リンク本数（本文2〜6本・重複なし＝`internal_links.md`設置ルール）／画像srcの絶対URL（ローカルパス・プレースホルダ残置の検出）。
**完了報告の前に「ERROR 0件」を確認する。WARNは公開前チェックリスト（下記8）として扱う。**

### PowerShellスクリプト作成時の注意（この環境固有）
- Windows PowerShell 5.1 はBOMなしUTF-8の`.ps1`をShift-JISとして読むため、**日本語をスクリプトに直接書くと文字化けで構文エラーになる**（実際に発生）。スクリプト本体はASCIIのみとし、日本語の検索パターンは `[char]0x8981` のようにコードポイント連結で構築する
- ファイル読み込みは常に `-Encoding UTF8` を付ける

## 7. JSON-LD（構造化データ）の配置

- `<!-- wp:html -->` ブロックの中に配置
- スクリプトタグの直前に `<!-- MANUAL_JSONLD -->` という目印コメントを入れる（プレーンなコメントなので3-1のような入れ子問題は起きない）
- 生成仕様は `structured_data_prompt.md` を参照

## 8. 公開前チェックリスト（WARNの解消）

ドラフト完成＝公開可能ではない。WordPressへ貼り付けて公開する前に以下を必ず解消する。

- [ ] 仮リンク `href="#"` を実URLに差し替え（またはリンクごと削除）
- [ ] JSON-LD内の「要修正」プレースホルダー（カテゴリURL等）を実データに差し替え
- [ ] JSON-LDの `dateModified` を実際の公開・更新日に設定
- [ ] 代表画像の実寸（1200x675推奨）を確認
- [ ] 投稿タイトル欄・SEOプラグインのメタディスクリプション欄へ、ドラフト冒頭の参照メモから転記
- [ ] 貼り付け後、ビジュアルエディターで「無効なコンテンツ」警告が出ていないか目視確認
