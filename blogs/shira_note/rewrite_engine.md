# shira_note リライト共通エンジン（速報型）

全番組共通の**手順骨格**。番組固有の値（タイトル形式・Grepマーカー・固定FAQ・見逃し配信等）は各番組パラメータファイル（`.claude/commands/{番組}-rewrite.md`）が持つ。

**適用範囲と優先順位**：
- 対象＝冒頭に「ENGINE準拠」宣言がある番組パラメータファイル（現在: utacon）
- **未移行の番組**（コマンドファイル内にフル手順があるもの）は、従来どおりコマンドファイルの手順を正とし、本エンジンは読まなくてよい
- ルールの重複時は「rewrite_common_rules.md ＞ 本エンジン ＞ 番組パラメータ」ではなく、**番組パラメータの固有規定が本エンジンの一般規定に優先**する（共通ルールは常に最上位）

---

## 0. 前提の確認

- 資料①（放送日・出演者・楽曲・見どころ）／資料②（直前回データ）／資料③（視聴メモ・任意）／アイキャッチURLを受領する。資料②が自由形式の場合の整形・承認手順は `CLAUDE.md`（shira_note直下）の資料②ガードレールに従う
- **資料②の時間整合チェック**：資料②の時間帯データが資料①の放送尺・放送時間と矛盾しないか確認する。矛盾時の処理（資料①優先か・ユーザー確認か・丸め方）は番組パラメータの規定に従う
- **通常回か特番かを資料①受領時に必ず確認する**（全番組共通の前提・2026-07-15確定）。歌唱曲・出演順の事前発表は「特番・SP回＝一部発表されることがある／通常回＝基本的に事前発表されない」。存在しない「◯月◯日発表予定」等の日付を創作して書かない
- **出演者未発表の場合**の共通方針：

| セクション | 扱い |
|---|---|
| aside・最新タイムテーブル・出演者一覧・出演時間予測・順番の傾向 | 「出演アーティスト未発表・判明次第更新」のプレースホルダー文言に差し替え（個別カードは削除し通知ブロック1枠に） |
| 実際の出演順（手順8） | 資料②を「直前回」として掲載（通常運用と同じ） |
| 過去のタイムテーブル一覧（手順10） | 通常運用どおり実施 |
| FAQ・まとめ・JSON-LD | 断定記述を避け「時点で未発表・判明次第追記」に統一。`mentions`/`performer`/`itemListElement` は空配列 |

発表され次第、同コマンドを再実行してフル更新に戻す。

## 1. 作業方針（Read/Edit規律）

| 原則 | 内容 |
|---|---|
| **WriteしないでEditする** | 変更箇所のみEditで差分更新。Write（全体上書き）は使用しない |
| **静的セクションはスキップ** | ナビゲーション/アフィリエイト等、番組パラメータで「固定」指定のセクションはRead・Edit不要 |
| **ドラフト全体を読まない** | Grepマーカー（番組パラメータの表）で行番号を特定し、必要範囲のみRead。全文Readは最終手段 |
| **ショートコードは独立ブロック** | `[nopc][title]`・`[nopc][mokujimae]`・`[kanrenad]`（旧`[nopc][originalsc][/nopc]`）は各々独立した `<!-- wp:paragraph -->` ブロックに分割し、前後テキストと混在させない |
| **大きなHTMLブロックのEdit** | 数十行の `old_string` は直前のRead結果からそのまま使う（記憶・手打ち再現はタイポで一致失敗する）。Edit失敗時は該当箇所を再Readして再試行 |
| **独立セクションは並行Edit** | old_stringが重複しないEditは1メッセージに束ねる。**JSON-LD（手順14）だけはFAQ確定後の最終バッチ**（一字一句一致のため） |
| **ユーザー修正後の確認** | system-reminderの差分 → 無ければ `git diff drafts/draft_{番組}.txt` を優先参照（詳細は `rewrite_common_rules.md` 9章） |

## 2. リライト手順（骨格）

### 1. 過去一覧の引き継ぎデータを Grep → Read で抽出
`過去のタイムテーブル一覧` をGrep → ヒット行から Read offset/limit で抽出。目的：過去一覧のカード群を次回ドラフトへ引き継ぐ。

### 1'. 上書き前の「実際の出演順【放送後更新】」を Grep → Read で抽出（退避データ）
`実際の出演順` をGrep → 同様に抽出。**この退避データは今回の資料②ではなく、資料②で上書きされる「前々回」のデータ**。手順10で過去一覧に追加するのは1'のデータであり資料②ではない。**1'と資料②を取り違えないこと**（2026-07-18 Mステの実ミスを受け全番組共通で明文化。取り違えると過去一覧とアーカイブパネルの内容が重複する）。

### 2. タイトル・メタディスクリプション
形式は番組パラメータの定型に従う。共通規律：
- タイトル・メタ設計の禁止事項は `rewrite_common_rules.md` 3章（誇張語・具体数値等）
- メタは120字以内。「（XXX字）」表記は実文字数と一致させ、候補提示前にPowerShellの`.Length`で一括実測（9章）
- JSON-LDの `description` はメタと一字一句一致（手順14で最終確認）

### 3. 冒頭段落・asideカード・締め文
番組パラメータのプレースホルダー表に従い更新。主観形容詞（「豪華」等）は使わない（共通ルール1章）。
**【SEO】** 締め文は「推しの出番を逃したくない」検索者（ペルソナ：佐藤 真由）の動機に直接応える、今回固有のコピーにする（汎用文禁止）。

### 4〜7. 本文H2群（タイムテーブル／出演者一覧／出演時間の目安／順番の傾向）
番組パラメータのセクション構成表に従い、今回の情報で更新する。
- 未発表項目は番組パラメータの統一文言を使い、曲名・発表日を創作しない
- **【SEO】** 説明文は「今回の顔ぶれだから起きること」を根拠に書く。汎用的な傾向文だけで済ませない

### 8. 実際の出演順【放送後更新】
**放送前は直前回（資料②）のデータを暫定掲載する。** 日付・出演者・タイムライン・注記を資料②に差し替え、放送後にリアルタイムで実データへ書き換える運用。

### 9. 見逃し配信・再放送
番組パラメータの規定に従う（「固定・スキップ」指定ならRead・Edit不要。文言指定があれば断定表現・サービス名の誤記に注意）。

### 10. 過去のタイムテーブル一覧

| 操作 | 内容 |
|---|---|
| 追加 | **手順1'で退避したデータ**を先頭に追加（資料②のデータではない。形式は番組パラメータ） |
| 維持 | 2回前の回はそのまま維持 |
| 移動 | 3回前の回を `<details>` 折りたたみ先頭へ移動（コンパクト形式に変換） |
| 削除 | `<details>` 内で最も古い回を削除 |

**完了後セルフチェック**：新規追加カードの放送日と手順8の放送日が同じになっていないかGrepで確認（同日付が2箇所＝1'と資料②の取り違え）。

### 11. よくある質問（FAQ）
番組パラメータのQ構成に従う。**固定指定のQは書き換えない**。
**【SEO】** Q文は読者が検索バーに打ち込む口語で書き、A文は冒頭1文で結論→補足の構造にする。

### 12. まとめ
放送日・出演者・見どころを今回の情報で書き換える（必須要素は番組パラメータ）。

### 13. ナビゲーション・アフィリエイトブロック（固定）
**スキップ（Read・Edit不要）。** URL一覧との整合確認は新番組追加時のみ（`rewrite_common_rules.md` 8章）。

### 14. 構造化データ（JSON-LD）— 最終バッチで実行

共通の差し替え箇所：

| 項目 | 更新内容 |
|---|---|
| `headline` / `description` | タイトル／メタと同一文字列 |
| `dateModified` | 更新日時（ISO形式 YYYY-MM-DDTHH:MM:SS+09:00） |
| `image` / `thumbnailUrl` | 今回のアイキャッチURL（未確定時の扱いは番組パラメータ） |
| `mentions` / `performer` / ItemList `itemListElement` | 出演者を最新に更新 |
| BroadcastEvent `name` / `description` / `startDate` / `endDate` | 番組パラメータの形式で今回の放送情報に更新 |
| ItemList `name` / `numberOfItems` | 番組パラメータの形式・出演者数に更新 |
| FAQPage `mainEntity` | 本文Q&Aと**一字一句一致**（固定Qは固定文のまま） |
| `keywords` | 番組パラメータの規則に従う |

固定箇所（`datePublished`・`author`/`publisher`・`about`・`BreadcrumbList` 等）は番組パラメータのリストに従い変更しない。

**アーティストの型判定**：グループ・バンド＝`MusicGroup`／ソロ＝`Person`。判定は `tools/generate_jsonld.py` のルールに準拠し、迷う場合は同ツールを使う。

## 3. 保存・共通チェックリスト

保存先：`drafts/draft_{番組}.txt`（Editで差分更新）。JSON-LDブロックはEditで一括差し替え。

保存後、番組パラメータの固有チェック行に加えて以下を確認する：

| 確認ポイント |
|---|
| メタが120字以内で、放送日・注目ポイント・更新訴求を含むか |
| 日付・出演者・楽曲・アーカイブの古い情報がすべて更新されているか（情報の鮮度） |
| ナビゲーションブロックが維持されているか（確認のみ・Edit不要） |
| JSON-LD FAQが本文と一字一句一致しているか |
| **最終実行**：`python tools/qa_draft.py draft_{番組}.txt` で ERROR 0件・新規WARN 0件（終了コード0） |

## 4. 表現多様化・視聴メモ（全手順に適用）

`rewrite_common_rules.md` **10章**を全手順に適用する：機能フレーズはローテーション（同一文3回以上はqa_draftがWARN検知）／資料③がある回のみ `class="shicho-memo"` ブロックを挿入（無い回に感想を創作しない）／独自予測はクッション言葉でデータと主観を区別（10-3）／完了報告に「この回の独自要素」1行（10-4 HCSセルフチェック）。
