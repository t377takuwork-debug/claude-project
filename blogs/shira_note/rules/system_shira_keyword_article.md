`/shira-keyword-article`（`blogs/shira_note/.claude/commands/shira-keyword-article.md`）で起動する、キーワード起点の新規テーマ記事制作フロー。2026-07-03新設。

**背景**: 「音楽の日 DREAMダンス」のように、番組タイムテーブル速報draft（`draft_XXXX.txt`）とは別枠で、特定キーワードの上位表示を狙う単発テーマ記事を作りたいという要望から新設。既存の番組リライトコマンド群はタイムテーブル速報専用で、先頭H2固定など構成原則が異なるため独立させた。

**フロー**: Step0 資料インプット確認（後述） → Step1 WebSearchで関連キーワード・共起語・想定検索キーワードを調査 → Step2 検索意図・ペルソナ確認（AskUserQuestion） → Step3 タイトル設計（2段階：分析フェーズでキーワード拡張/ペルソナ2パターン/検索意図を提示しユーザーの「OK」を待ってから10パターンのタイトル案＋キーワード使用チェックを出力） → Step4 見出し構成設計（構造化アウトラインH2/H3＋メタディスクリプション案＋スラッグ案＋設計意図メモの4点セット） → Step5+6 本文執筆と構造化データ生成を**1回のWriteでまとめて**`draft_kw_{slug}.txt`に保存 → Step7 `qa_draft.py`を**確認を挟まず自動実行**し、ERROR/WARNがあれば都度自己修正してからERROR 0件の状態のみユーザーに報告。

**Step5〜7の効率化**（2026-07-03追加、`draft_dancebattle.txt`の実セッションを踏まえた改善）: 当初は本文保存→JSON-LD追記→「検品しますか？」とユーザー確認→検品→エラー修正、という順に複数ラウンドに分かれていたが非効率だったため、①本文と構造化データは1回のWrite/Editでまとめて保存する、②検品はユーザーに確認を取らず保存直後に自動実行し、ERROR/WARNが出れば黙って直してから完了報告する、という2点に変更。

**Step0資料インプット確認フェーズ**（2026-07-03追加）: WebSearchは検索結果の要約スニペットしか取れず、公式サイト詳細ページの粒度細かい情報（事務所別メンバー個別名等）をほぼ拾えないという精度限界が判明（「音楽の日 ダンスバトル」事例で、WebSearchでは主要事実しか拾えず、ユーザーが独自調査した公式サイト全文情報を提供して初めて正確なチーム編成が書けた）。このためStep1でWebSearchを走らせる前に、必ずユーザーに「一次資料はお持ちですか」と確認するフェーズをStep0として新設。資料が無い場合のみWebSearchに進むが、個別名等の細かい粒度は限界がある旨を先に伝える。

**Step6構造化データ生成フェーズ**（2026-07-03追加、出所は`draft_dancebattle.txt`で確立した形式）: 本文末尾に`<!-- wp:html --><!-- MANUAL_JSONLD --><script type="application/ld+json">`形式で`@graph`統合のJSON-LDを配置。固定データ：publisher/author共に"Shira Notes"（sameAs `x.com/ShiraNotes_`、operator-information URL）。画像は1200×675固定。`about`は必ず`Thing`型（Event/Product/Person禁止）。`BroadcastEvent`は`location`未公表時`VirtualLocation`で代替、`performer`は本文記載分のみ反映（推測禁止）。`BreadcrumbList`はカテゴリURL未確定なら「ホーム→記事→本記事」の3階層（`draft_ongakunohi.txt`と同じ簡易パターン）。メタディスクリプションと`description`の一字一句一致が`qa_draft.py`のERROR対象。

**Step3タイトル手法の出所**: `c:\Users\PC_User\OneDrive\デスクトップ\音楽プロンプト\タイトル.txt`（2026-07-03、ユーザーが持ち込んだ既存プロンプトを恒久的にスキルへ統合）。10パターン＝汎用/CTR特化/AI検索向け質問型/速報性/総まとめ/Discover向け/企画内容推し型/出演者ファン向け/ライト層向け/ニッチ検索拾い型。タイトル長32〜40文字目安、記号は10本中4本以上、主観的感想・スラング・上から目線表現は禁止。

**Step4見出し構成手法の出所**: `c:\Users\PC_User\OneDrive\デスクトップ\音楽プロンプト\見出し構成.txt`（2026-07-03、同様に恒久統合）。最低1つのH2は質問形／強い検索ニーズ（出演者は誰か等）に見出しレベルで直接応える／未発表情報は「現時点で分かっている範囲」等の安全な表現／FAQはまとめ直前・H3で3〜4問／まとめにH3は付けない／「なお」「一方で」「本記事では」等の硬い接続語の多用を避ける。

**Step5本文執筆手法の出所**: `c:\Users\PC_User\OneDrive\デスクトップ\音楽プロンプト\リード文.txt` と `文章の型.txt`（2026-07-03、恒久統合）。リード文3〜5文・110〜150字・メインキーワードを1〜2文目に含める。H2単体最大400字/H3は220〜280字目安。未確定情報は「〜と考えられます」等で断定を避ける。**ユーザー指定の調整**：①H2ごとの分割出力＋都度確認は行わず通しで執筆する、②各H2への視覚要素（表/箇条書き等）の必須化はしない。③クイック情報ボックス等の`<aside>`/`<div>`装飾HTMLは固定テンプレートではなく記事ごとに臨機応変にデザイン生成する（`wp:html`で囲む・`role="region" aria-label`必須・レスポンシブ対応必須、という技術ルールのみ固定）。2026-07-03時点でこの方針に更新（当初「本文に含めない」としていたが、実際のセッションで臨機応変な生成が採用されたため）。

**How to apply**: ユーザーが「〇〇というキーワードで記事を作りたい/上位表示を狙いたい」と言ったら、このコマンドを案内・起動する。SEOの構成原則は`blogs/seo/SEO_guide.txt`が唯一の正。捏造禁止ルールはこのフローにも適用（事実不明な事項はユーザーに資料提供を求める）。

**毎年恒例の企画・コーナーが対象の場合の追加ルール**（2026-07-03「音楽の日 ダンスバトル/DREAMダンス」事例で確定）:
- Step1で必ず「名称が年により変わっていないか」を確認する。公式の現行名称よりも旧称・通称の検索需要が大きいことがある（例：2026年の公式名称は「DREAMダンス」だが、通称「ダンスバトル」の方が検索ボリュームが大きく、主キーワードは「ダンスバトル」＋「DREAMダンス」併記とした）
- 自社サイト（shira-treat.com）に同一企画の過去記事が無いかWebFetchで確認し、あればURLスラッグ命名パターン・見出し構成を踏襲する
- ファイル名は`draft_kw_{slug}.txt`ではなく、番組特番draft群と同じ**年号なし固定ファイル名**（例：`draft_dancebattle.txt`）にし、来年以降も上書きリライトする。CLAUDE.mdの「ドラフトファイルの命名規則」表に追記する
- 公開完了後のdraftファイルはローカルに蓄積し続ける必要はない（本体はWordPress側が正）。ただし毎年恒例コーナーは次回リライトのベースとして残す
