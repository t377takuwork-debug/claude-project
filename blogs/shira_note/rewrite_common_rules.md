# shira_note 全リライト共通ルール集

全番組のリライト・新規記事作成で必ず適用するルール。各`*-rewrite`コマンド実行時はこのファイルを最初に読むこと。
（旧: メモリ13ファイルに分散していたルールを2026-07-05に統合。以後このファイルが唯一の正）

多くの項目は `python tools/qa_draft.py`（`/shira-qa`）が機械チェックするが、**生成時点から守ることで手戻りをなくす**のが目的。

---

## 1. 禁止表現・ワード（本文・カード説明文・まとめ文・冒頭締め文すべて）

| 禁止表現 | 代替の方向性 |
|---|---|
| 幕を開けます | 「続く」「登場する」「スタートする」等、動作で言い換える |
| 筆頭に、 | 並列で事実を並べる（「〜のコラボ、〇〇・△△など〜が」） |
| 見せ場です | 「集中する」「登場する予定」等、事実ベースの記述に |
| フィナーレを飾ります／締めくくります | 「ラストに登場します」「最後に登場します」等（「担います」はかしこまりすぎなので使わない） |
| 彩る／彩ります／飾る | 「担当する」「出演する」「登場する」等、具体的な動詞に（「集結」も同様の理由で使わない。下段参照） |
| 集結 | 「出演する」「登場する」「顔ぶれが揃う」等に言い換える（2026-07-17追加） |
| 感動の〜／〜が大きな見どころです等の抽象的な形容 | 「感動の」「圧巻の」等の主観的な修飾より、具体的な事実（曲名・出演者・時間帯等）で語る（2026-07-17追加） |
| 編集部 | 「筆者」「当サイト」に言い換える（一人運営のため、組織を装う表現はE-E-A-T上マイナス。2026-07-16追加） |

テンプレ感・AI文体が出やすい表現のため全番組で禁止（2026-07-01確定）。

**未確定の順番・位置を断定しない**：「番組フィナーレ」「トリを飾る」等、放送内での位置づけを断定する表現は、資料上その順番が実際に確定している場合のみ使う。時間帯・出演者・内容は判明していても、時間帯内の細かい出演順までは分からないケースが多いため、「〜が登場します」等の中立的な表現に留め、確定していない順序を断定しない（2026-07-17 音楽の日リライトでユーザー指摘により明文化）。

## 2. 段落・文体ルール

- **1段落 = 1テーマ**。複合文（「〜が発表されました＋状況説明」等）は生成時点から別々の`<!-- wp:paragraph -->`ブロックに分ける
- アーティスト列挙文と「今後も追加発表予定」等の告知文も原則別ブロック
- 冒頭リード文（`[nopc][title]`直前）は**1文のみ**（テーマと規模感だけ。「随時更新」等の更新告知は入れない）
- 冒頭2段落目は放送形式の変更点のみ（MC情報・見どころは混ぜない）
- タイムテーブルH2直後リード文は1文のみ（「総勢〇組」「随時更新」不要）
- 締め文（`[nopc][mokujimae][/nopc]`直後）に**主観形容詞NG**（「感動的な」「豪華な」「圧倒的な」等）。「企画名＋など見どころ満載！」の形で完結
- 締め文に具体企画名の詳述は不要。「更新・ブックマーク」の促進のみ
- H2見出しの「｜補足サブテキスト」は、本文に同内容があれば省略を検討（見出しは短いほど良い）
- 資料①に含まれる周辺トリビア（異体字・旧字体の由来、結成◯周年、一部公演への不参加情報等）は、フッター注記やカード説明文などスペースが限られた箇所に機械的に全部詰め込まない。情報量過多で視認性が落ちるため、追加するか・どこに置くか迷ったら最小限（触れない）に倒すか一言確認する（2026-07-08 うたであえたら回でユーザーが差し戻し）
- 冒頭aside「番組情報まとめ」のステータス表示は、数値のみ（「31組発表」等）より代表アーティスト名2〜4組を例示した形式（「あいみょん、くるり、福山雅治、LUNA SEA他、全32組」等）が好まれる。ラベルも「出演者：」→「主な出演：」、「進行：」→「タイムテーブル：」のように内容が一目でわかる語を優先する（同上セッションでの手直し）

## 3. タイトル・メタディスクリプション

- **数値を焼き込まない**：「31組」「司会3名」等その時点の具体数値はタイトル・メタに入れない（追加発表のたびに陳腐化するため）。年号は「番組名2026」のように名称へ直接付け、末尾の【2026】ブラケットは使わない
- **誇張語NG**：「どこよりも早く」「最速で」等の速報訴求はトーンに合わない。「タイムテーブル」「出演者」「順番」等の検索KWと更新中の事実記述に留める
- **番組テーマ名を入れない**（例:「こえる。きこえる。」）。メタ末尾は「タイムテーブルや出演者の順番を最速でお届けします」等KW直結の締めにする
- **メタディスクリプションに「出演順」「順番」「曲順」「登場時間」のいずれかを必ず含める**（タイトルには標準で入っているがメタに漏れやすい。2026-07-11 STARリライトでメタのみ抜けていたユーザー指摘により明文化。既存の全番組ドラフトは満たしている）
- 字数オーバー時（目安120字以内）は修飾語を削ってよい（例:「SUPER BEAVERのスペシャルステージ中継」→「SUPER BEAVERの中継」）
- **メタを修正したらJSON-LDのdescriptionも一字一句同じ文に更新する**（headlineも同様。ユーザー手直し後は聞かれる前に再同期を確認）

## 4. WPブロック構造

- **ドラフト冒頭のタイトル・メタ行も`<!-- wp:paragraph -->`〜`<!-- /wp:paragraph -->`で囲む**（参照メモ扱いで除去しない。ユーザー指定仕様）

```
<!-- wp:paragraph -->
タイトル：{タイトル文字列}
メタディスクリプション（n字）：
{メタディスクリプション文字列}
<!-- /wp:paragraph -->
```

- **ショートコード（`[nopc][title]`・`[nopc][mokujimae]`・`[nopc][originalsc]`）は必ず単独のwp:paragraphブロックに配置**。前後テキストとの同一ブロック混在は絶対NG
  - NG例: `<p>テキスト[nopc][title][/nopc]</p>` / `<p>[nopc][mokujimae][/nopc]締め文</p>`

## 5. HTML実装ルール

- **インラインスタイル付き`<ul>`は使わず`<div>`で書く**（WPテーマCSSがfont-size等を上書きするため）
  - `<ul style="...">` → `<div style="...">`（list-style等リスト固有CSSは削除）、`<li>` → `<div>`、シンプルな箇条書きは先頭に`・`
- **スマートクォート禁止**：EditでHTMLブロックを差し替える際、U+201C/U+201D/U+2018/U+2019が混入するとHTML属性が壊れる。リライト完了後に必ず`python tools/qa_draft.py {ファイル名} --fix`で検証・修正

## 6. 継続性表現の誤読防止

- 「〇年連続」「引き続き」「再び」等を使う場合、直前直後に対象番組名を明記し、今回の番組とは別物だと一文で分かるようにする（年1回特番は「今年が初回」が根幹事実のため）
- 本文修正時は、同内容を要約したバッジ・見出し・カード内テキスト・JSON-LDにも同表現が残っていないか**Grepで横断チェック**（「昨年」「引き続き」「連続」等）

## 9. 効率化ルール（Read/Edit・往復削減）

- **ユーザー手動編集の確認はgit diffを優先する**：ドラフト修正後にユーザーが直接編集した内容を確認する際、system-reminderに具体的な差分が含まれていない場合は、まず `git diff {ファイル名}`（PowerShell）で変更行のみ取得する。全文Read（複数チャンクにまたがる場合）での目視比較は、トークン消費が大きく見落としリスクも高いため避ける。gitで追跡されていない・差分取得に失敗した場合のみ全文Readにフォールバックする。
- **文字数を伴う複数候補は提示前に一括実測する**：タイトル・見出し・メタディスクリプション等、文字数の優劣で判断が必要な提案をユーザーに出す前に、必ずPowerShellで全候補の`.Length`を実測してから提示する。目算・概算での字数記載は誤りの原因になり、訂正の往復が発生する（2026-07-17 Mステ1500回記念H2で目算「35字前後」が実測42字と判明し手戻りが発生）。

## 10. 表現多様化・視聴メモ・予測の書き方（Experienceシグナル強化・2026-07-16追加）

12本横断分析（`tools/output/analysis_template_expressions.md`）で、量産型に見える主因は「機能フレーズの定型化」と「リライト間の文面94%同一」にあると判明した。本章はその対策。SEO構成（H2順・タイトル/メタ設計・JSON-LD）は従来どおり維持し、**表現レイヤーだけ**を多様化する。

### 10-1. 機能フレーズのローテーション

リライトで文言を触る箇所では、以下の高頻度フレーズを**前回と同じ言い回しで書かない**（前回の文言は現draft内をGrepすれば確認できる）。表は起点であり、この場で新しい言い換えを作ってよい（意味・事実を変えないこと）。

| 機能 | 使い回しがちな型 | バリエーション例 |
|---|---|---|
| 更新予告 | 「判明次第更新します」 | 分かり次第ここに反映します／発表が出たらすぐ書き足します／解禁を待って追記予定です／放送直前にもう一度更新をかけます／（放送中）リアルタイムで追いかけながら更新中です |
| 未発表案内 | 「現時点で未発表です」 | まだ発表が出ていません／今のところ公式からのアナウンスはなし／発表待ちの状態が続いています／例年どおりなら発表は放送◯日前ごろです／公式の解禁はもう少し先になりそうです |
| まとめ導入 | 「〜をまとめました」 | 〜を一覧にしています／〜を整理しました／〜をこのページに集約しています／〜が一目で分かるようにしました／〜を時系列で追えるようにしています |
| CTA | 「ぜひブックマークして〜」 | 当日また覗きに来てもらえたら嬉しいです／放送前にもう一度チェックしに来てください／このページを開いたまま観るのがおすすめです／気になる方は保存しておいてください／出番が近づいたら見に来てください |
| 傾向言及 | 「〜という傾向があります」 | 過去◯回の放送では〜が続いています／ここ数年は〜というパターンです／〜になることが多い印象です（※10-3のクッション言葉と併用）／直近◯回を見る限り〜です |

- **同一文を1記事内で3回以上リピートしない**（`qa_draft.py` がWARN検知。数字だけ違う文は同一とみなす）。注意書き等を複数箇所に置く場合は、2箇所目以降を言い換えるか「前述のとおり〜」で参照する
- 各記事のリード文・締め文は既に番組ごとに個性がある（分析済み）。**リライト時にこれを他番組の文面に寄せない**

### 10-2. 視聴メモブロック（資料③）

著者が実際に番組を見ている実体験（Experience）シグナルを入れる専用ブロック。

- **資料③（ユーザーが渡す視聴メモ・注目点の断片メモ）がある回のみ挿入する。無い回はブロックごと省略する。AIがメモや感想を創作することは捏造にあたり禁止**
- メモ→本文化は、メモに書かれた事実・感想の範囲内で1〜3文に整える。誇張・事実の追加・他アーティストへの一般化をしない
- マークアップ（`class="shicho-memo"` は必須＝qa_draft連携の目印。**ブロック内に子divを置かない**）：


### 10-3. 予測・推測のクッション言葉（データと主観の区別）

## 11. セクションの役割分担・重複排除・FAQ設計（2026-07-17追加・全番組共通）

Mステ記事のFAQ会議（/blog-kaigi）とセクション棚卸しで確立した考え方。記事構造は番組ごとに異なるが、以下の原則は全番組のリライト・新規記事で適用する。**各番組draftの構造変更（カード削除・セクション再編等）を伴う場合は、実施前にユーザーへ提案して承認を得る。**

### 11-1. 同一情報の掲載回数を管理する（重複排除）

- 見どころ・出演者列挙・放送時間などの核情報は、**役割の異なる2〜3箇所まで**（例：冒頭aside＋該当H2＋まとめ）。リライト中に4箇所目が生まれそうなら、どこかを削るか役割を分担し直す
- 各セクションには見出しの主題に沿った情報だけを置く（例：「順番の傾向」H2に見どころ列挙を置かない）。主題外の文は有益でも削除候補として扱う
- 企画コーナーの候補曲リストなど同じ詳細情報は1箇所（出演者・歌唱曲一覧のカード側）にのみ置き、他セクションでは要点・時間帯情報だけに触れる
- 直前回の放送データは「実際の出演順」アーカイブパネル（該当番組にある場合）が正の置き場所。過去一覧セクションに同じ回のカードを重複して置かない（Mステは2026-07-17にフルカード全廃・details一本化済み）

### 11-2. 冒頭asideの行ごとの役割分担

- STATUS＝情報公開ステージ（出演者発表→歌唱曲発表→放送中→放送終了）を短く
- TIMETABLE＝タイムテーブルの公開状況と更新方針
- 💡コメント＝見どころの要約（「今回だからこそ見る理由」はここに集約）
- mokujimae直後の締め文＝更新・ブックマーク促進の1段落のみ。asideの要約と同内容の見どころ段落を重ねない

### 11-3. FAQ設計の3基準と枠の定式（/blog-kaigi 2026-07-17決定）

FAQに置く質問は、以下の3基準をすべて満たすこと。

1. **実際に検索バーに打ち込まれる言葉か**（「〜は何時頃？」「〜はありますか？」等の口語疑問形。FAQPageリッチリザルトは期待せず、PAA・ページ内の検索意図カバーを目的とする）
2. **その答えを求める読者が実在するか**（「なぜ出演者が多いのか」のような編集側の説明はNG）
3. **根拠を持って答えられるか**（一次情報の無い質問はFAQに入れず、判明時に本文へ追記して対応する）

速報型記事の枠の定式（デフォルト。年1特番型は番組特性に合わせて調整可・上記3基準は必須）：

- Q1＝タイムテーブルの事前発表（定型。回答に放送枠「HH:MM〜HH:MM」を明記し「何時から何時まで」の検索意図を吸収）
- Q2＝その回の最大目玉アーティストの出番予測
- Q3＝動員力最大のファンダムを持つグループの出番予測
- Q4＝見逃し配信の有無（定型）
- Q5＝番組固有の固定質問（エイリアス伝達等・編集不要）

回答の原則：**結論先出しの1〜2文**。出演者の全列挙など本文と重複する長い列挙は入れない。予測系の回答には根拠（過去の傾向。10-3のクッション言葉と併用）と「実際の出演順は放送後に本ページで更新」の再訪導線を添える。

## 12. レスポンシブ文字サイズ・レイアウト（PC/スマホ両対応・2026-07-17追加）

カード型HTML（タイムテーブル・出演者グリッド・見どころボックス等）はスマホでは見やすいが、PCで表示すると本文（H2見出し・段落）に対して文字が小さく感じられることがある（2026-07-17 音楽の日リライトでユーザー指摘）。

- **font-sizeは`clamp()`で書く**：`font-size: clamp(最小px, calc(最小px + (100vw - 480px) / 除数), 最大px)`。最小値＝現状のスマホ表示（変更しない）、最大値＝PCで見やすい大きさ（目安：元の値の1.3倍前後）。除数は `(1200 - 480) / (最大 - 最小)` を目安にし、画面幅1200px前後で最大値に到達するようにする
  - 480px以下の画面幅では計算結果が最小値を下回るため、clampが自動的に最小値を返す＝スマホ表示は無変更
- **固定幅の狭いカラム**（タイムテーブルの時間ラベル列等）も同じ考え方で`width`をclamp化し、文字拡大に対して窮屈にならないようにする。あわせて`white-space: nowrap`を付けて折り返しを防ぐ
- 新規記事・新番組立ち上げでは、カード型HTMLを書く時点からこの形式をデフォルトにする。既存ドラフトのリライトでは、触っている範囲だけ適用すればよい（未着手箇所への一斉適用は不要）

## 13. 出演者一覧・企画詳細の精度ルール（2026-07-17追加）

複数事務所・複数アーティストが参加する合同企画（ウタウタイ・DREAMダンス・DREAMコーラス等）を扱う際の精度ルール。

- **グループ名と個人名の表記統一**：グループの一部メンバーのみが出演する場合は、グループ名を主表記にし、個人名は歌唱曲と同じ見た目のサブテキストで「〇〇が出演」と示す。個人名を主表記にして「（グループ名）」と続ける書き方はしない（意図が逆になり誤解を招く）
  - 例外：本人が個人名義で活動している場合（ソロデビュー済みメンバー等）は個人名を主表記にしてよい。判断に迷ったら資料の書き方（グループ名が先に来ているか個人名が先か）に従う
- **タイムテーブルは全出演者ボックスに歌唱曲を反映する**：メドレー・企画コーナーの目立つボックスだけでなく、通常出演枠の1件ずつにも資料にある歌唱曲を反映する。ハイライトボックスにだけ曲を入れて通常枠を素通ししやすいので、リライト完了前に全ボックスを見比べて確認する
- **合同企画の参加者リストは最新の資料バージョンと必ず突き合わせる**：資料が複数回に分けて提供される場合、後から提供された資料がより正確な最終版であることが多い。参加者リストを転記する前に、旧版との差分（追加・削除された名前・所属の修正）を確認する
- **注記（※〜）は該当する項目の直下に置く**：個別の企画に関する補足説明は、その企画の直後に配置する。複数企画の注記をまとめて記事末尾に置かない
- **詳細情報が別記事にある場合は簡易サマリー＋リンクに留める**：企画の内訳（事務所別ラインナップ等）が独立した詳細記事を持つ場合、本編には簡易な概要とテキストリンクのみ置き、全内訳を重複掲載しない

## 14. 見どころセクションのH3設計方針（2026-07-17追加）

- 「見どころ」H2内の各コーナー・企画は、原則としてH3見出しにしない（小見出しが増えすぎるとページの構造が過度に細分化される）。H3にするのは、そのH2配下の主要な塊（例：「番組テーマ」「特別企画・見どころ一覧」）までに留める
- 個別の企画名（EXILE南三陸ライブ、DREAMダンス企画等）は、太字ラベル＋詳細ボックスの組み合わせで表現し、通常の見出しタグは使わない
- 内訳テーブルの列数はスマホでの折り返しを避けるため2列を基本とする（1列目の項目名セルに`white-space: nowrap`と概ね30%前後の`width`を指定し、2列目に詳細をまとめる）。3列以上にする場合は、1列目が折り返さないか必ずスマホ幅で確認する
