# Mステ タイムテーブル記事リライト

Mステ（ミュージックステーション）のタイムテーブル記事を最新情報にリライトし、`draft_mste.txt` として上書き保存する。

## 事前情報の確認

### 必須情報（資料①）
- 放送日（YYYY年MM月DD日）
- 放送時間・チャンネル
- 出演アーティスト一覧
- 各アーティストの披露曲
- 今回の見どころ・注目ポイント（コラボ・初披露・特別企画など）
- VTR企画の有無と内容
- 特別編成の有無（〇〇回記念・年末SP・コラボ企画等の名称。通常回の場合は不要）

### 推奨情報（資料②）
- 直前回のX（旧Twitter）投稿分析データ
  - 各アーティストの出演時間帯・順番
  - SNS反応のピーク・盛り上がりポイント
  - 「調査結果」として過去一覧カードに反映する

### その他
- アイキャッチ画像URL（構造化データ（JSON-LD）に反映）

### 資料①が「出演者未発表」の場合

次回放送の出演者がまだ発表されていない状態でリライトを依頼された場合（資料②のみ提供・資料①なし）、以下の方針で進める。

| セクション | 扱い |
|---|---|
| aside・最新タイムテーブル・出演者一覧・出演時間予測・順番の傾向 | 「出演アーティスト未発表・判明次第更新」のプレースホルダー文言に差し替える（個別アーティストカードは削除し、通知ブロック1枠にする） |
| 実際の出演順アーカイブパネル（手順8） | 資料②（直前回データ）をそのまま「直前回」として掲載する（通常運用と同じ） |
| 過去のタイムテーブル一覧（手順10） | アーカイブパネルの旧回データをdetails先頭へコンパクト形式で追加する処理は通常通り実施 |
| FAQ・まとめ・JSON-LD | 出演者名を含む断定的な記述を避け、「時点で未発表・判明次第追記」の文言に統一。JSON-LD の `mentions`/`performer`/`itemListElement` は空配列にする |

出演者が発表され次第、再度 `/mste-rewrite` を実行し、このプレースホルダー状態から通常のフル更新に戻す。

### 資料①に特別編成（〇〇回記念・SP企画等）がある場合

放送回が「〇〇回記念」「年末SP」「コラボ企画」等の特別編成の場合、通常回のテンプレートに加えて以下7箇所に特別回名を追加する（2026-07-17「1500回記念SP」対応で確立したパターン）。

| セクション | 追加方法 |
|---|---|
| タイトル | 「Mステ タイムテーブル【M月DD日】{特別回名} 出演順・出演時間をリアルタイム更新」の形で挿入 |
| メタディスクリプション | 冒頭固定句直後・注目ポイント列挙の先頭に追加 |
| 冒頭段落1 | 「{放送日}のMステは「{特別回名}」。」の形で明示 |
| asideカード 💡コメント行 | 冒頭に「{特別回名}の節目回。」等を追加 |
| FAQ（出演者数・編成理由に触れる質問があれば） | 回答内で「{特別回名}として」に言及 |
| JSON-LD headline/description | タイトル・メタと同期 |
| JSON-LD BroadcastEvent name/description | 番組名・説明文に特別回名を追加 |

**H2見出しへの追加は行わない**（見出しが長大化し他見出しとの均衡を崩すため。特別回名は本文側7箇所で十分に露出しており、見出しでの情報欠落にはならない）。

特別編成でない通常回では、上記箇所に特別編成の文言を一切加えない。**前回が特別回だった場合、通常回に戻す際は残存した特別回名の文言をGrepで検索し、取りこぼしなく除去する**（例：`Grep "1500回記念"` で全箇所を洗い出してから通常回テンプレートに戻す）。

## 作業方針

| 原則 | 内容 |
|---|---|
| **WriteしないでEditする** | 変更箇所のみEditで差分更新。Write（全体上書き）は使用しない |
| **静的セクションはスキップ** | 見逃し配信・ナビゲーション/アフィリエイトブロックはRead・Edit不要（ドラフトの既存ブロックをそのまま維持する。`template_mste_static.txt` はバックアップ参照用） |
| **ショートコードは独立ブロック** | `[nopc][title]`・`[nopc][mokujimae]`・`[nopc][originalsc]` はそれぞれ独立した `<!-- wp:paragraph -->` ブロックに分割する |
| **ユーザー修正後の確認** | system-reminderの差分 → 無ければ`git diff drafts/draft_mste.txt`（PowerShell）を優先参照し、JSON-LD FAQ整合性など必要箇所のみRead・確認する（全文Readは最終手段。詳細は`rewrite_common_rules.md` 9章） |
| **大きなHTMLブロックのEdit** | ARTIST LINEUPカード群・アーカイブパネルなど数十行にわたる `old_string` は、直前のRead結果からそのまま使う（記憶や手打ちでの再現はタイポで一致失敗しやすい）。Editが失敗したら該当箇所を再Readしてから再試行する |

| 出演者・歌唱曲一覧 | `ARTIST LINEUP` |
| アーカイブパネル | `SETLIST` または `ARCHIVE` |

| FAQ | `よくある質問` |
| JSON-LD | `MANUAL_JSONLD` |

## リライト手順

### 1. ファイル準備（①と②を並列実行）

以下を**同時に**実行する（順番待ちしない）。

**① 直前回ドラフトの「過去のタイムテーブル一覧」を Grep → Read で抽出**
```
Grep パターン: "過去のタイムテーブル一覧"
  → ヒット行番号を確認し、Read offset=<その行> limit=250 で抽出
```
取得目的：details内コンパクトカードのデータを次回ドラフトに引き継ぐため。

**③ 上書き前の「実際の出演順【放送後更新】」アーカイブパネルを Grep → Read で抽出（退避データ）**
```
Grep パターン: "SETLIST" または "ARCHIVE"
  → ヒット行番号を確認し、Read offset=<その行> limit=100 で抽出
```
取得目的：このアーカイブパネルの中身は**今回の資料②ではなく、今回の資料②で上書きされる「前々回」のデータ**。手順10で過去一覧のdetails先頭に追加するのはこの③のデータであり、資料②（今回アーカイブパネルに新しく入れるデータ）ではない。**③と資料②を取り違えないこと**（2026-07-18 Mステ7/24回でユーザー指摘により明文化。過去一覧に資料②の日付を入れてしまい、アーカイブパネルと内容が重複する誤りが発生した）。

**② WebFetch で YouTube 最新動画URL を取得（RSSフィード経由）**
- `https://www.youtube.com/feeds/videos.xml?channel_id=UCStFsybJM6SjDFvcacVcanw`
- チャンネルページ（`/videos`一覧URL）は動的レンダリングのためWebFetchでは中身が取得できないことが多い。RSSフィードは静的XMLで確実に取得できるため、必ずこちらを使う。
- 取得後、先頭（最新）エントリの `yt:videoId` を動画IDとして使用する。

### 2. タイトル・メタディスクリプション

**タイトル**
```
Mステ タイムテーブル【M月DD日】出演順・出演時間をリアルタイム更新
```
- 「Mステ タイムテーブル」を冒頭に固定
- 放送日を【】で表記

**メタディスクリプション（120字以内）**

ドラフトファイルのタイトル行の直下に以下の形式で記載する。タイトル行〜メタディスクリプション行は `<!-- wp:paragraph -->` 〜 `<!-- /wp:paragraph -->` で囲む（ユーザー指定仕様。参照メモだが除去しない）。

```
<!-- wp:paragraph -->
タイトル：Mステ タイムテーブル【M月DD日】出演順・出演時間をリアルタイム更新
メタディスクリプション（XXX字）：
Mステ（M月DD日）タイムテーブルを速報更新！{最大注目ポイント1}・{注目ポイント2}・{コラボや企画}。出演順・登場時間の目安を随時追記。
<!-- /wp:paragraph -->
```

> 文字数表記（XXX字）は実際の文字数と必ず一致させる（qa_draft.pyがWARNで検出する）。JSON-LDの`description`もメタディスクリプションと一字一句一致させる（手順14で最終確認）。
> 文字数カウントは **PowerShell の `$s.Length`** を使う（Bashツールはこの環境で出力が返らないことがあるため、最初からPowerShellを使う。詳細は memory の `project_bash_tool_output_broken.md`）。

#### 作成ルール

| 項目 | 内容 |
|---|---|
| 冒頭 | 「Mステ（M月DD日）タイムテーブルを速報更新！」で固定（鮮度シグナル） |
| 注目ポイント | 初披露・ラスト出演・コラボなど、この回固有の見どころを「・」区切りで列挙 |
| 末尾 | 「出演順・登場時間の目安を随時追記。」で固定（検索意図との一致） |
| 文字数 | 100〜115字を目安。120字を超えない |
| アーティスト名 | 最大注目の2〜3組を優先して入れる。全員を列挙しない |

### 3. 冒頭段落・aside カード

以下を更新する。

| プレースホルダー | 内容 |
|---|---|
| 放送日時 | YYYY年MM月DD日(曜) HH:MM〜HH:MM |
| STATUS | 情報公開ステージを短く記す（放送前：「出演アーティスト・歌唱曲 発表済み」／放送中：「放送中・出演順をリアルタイム更新中」／放送後：「放送終了・実際の出演順を反映済み」等。括弧書きの補足は付けない） |
| ARTISTS | 出演者を `/` 区切りで列挙 |
| TIMETABLE（公開状況） | タイムテーブルの公開状況＋更新方針を1〜2文で（例：「公式の分刻みタイムテーブルは未公開。出演順・時間帯は放送に合わせてリアルタイム更新します。」）。進行予想・見どころは書かない（見どころは「編集者コメント」行に集約。2026-07-17に役割変更） |
| 編集者コメント | 初披露・ラスト出演など特記事項（見どころの列挙はこちら） |
| 最終更新日 | 記事作成日（YYYY.MM.DD） |
| 「最新情報を確認する」リンク | href のアンカー（標準値 `#st-toc-h-1`）は本文最初のH2見出し（最新タイムテーブルセクション）へのTOCジャンプ用。前回値をそのまま踏襲せず、見出し構成に変更がないか都度確認する |

締め文（`[nopc][mokujimae][/nopc]` 直後）は更新・ブックマーク促進の1段落のみ（例：「推しの出番を逃さないよう、出演順やタイムテーブルの目安を放送に合わせて随時更新予定です。」）。**asideと同内容の見どころ要約段落は置かない**（重複のため2026-07-17に廃止）。主観形容詞（「豪華」等）は使わない（qa_draft.pyがWARNで検出する）。

> **ショートコードは独立ブロック**：`[nopc][title]`・`[nopc][mokujimae]`・`[nopc][originalsc]` は必ず独立した `<!-- wp:paragraph -->` に配置する。
> ❌ `<p>〜更新します。[nopc][title][/nopc]</p>`　✅ `<p>〜更新します。</p>` + 独立 `<p>[nopc][title][/nopc]</p>`

**【SEO】** 「今回だからこそ見る理由」はaside💡コメント行で1文で示す（ペルソナ：佐藤真由の「推しの出番を逃したくない」動機に直接応える）。締め文はその動機を「随時更新するからこのページを見ればよい」という安心につなげる役割に徹する。

### 4. H2：最新タイムテーブル
カード内のヘッダーの日付・出演者バッジを更新する。
予測の時間帯・注目ポイントの説明文は今回の情報に合わせて書き換える。

### 5. H2：出演者・歌唱曲一覧
各アーティストカードを最新の出演者・楽曲に差し替える。
`ARTIST LINEUP (YYYY.MM.DD)` の日付も更新する。

### 6. H2：Mステの出演時間は何時頃？【目安】
見出しの「Mステの」は固定（検索順位強化のための固定表記。変更しない）。
3ブロック（序盤・中盤・終盤）の説明を今回の出演者・見どころに合わせて書き換える。
カードヘッダーの日付（例：`6/12 出演時間帯の予測ガイド`）も更新する。

**【SEO】** 各ブロックの説明文は「過去の傾向＋今回の出演者を組み合わせた独自予測」として書く。「一般的に序盤は〜」といった汎用文ではなく、今回の顔ぶれを根拠にした具体的な予測にすることで一次情報としての価値を出す。

### 7. H2：Mステの順番の傾向
カードは**4枚構成**（🕒通常放送の構成／🌟トリの予測／🔍VTR企画のタイミング／📱公式SNSの連動）。「今週の注目パフォーマンス」カードは置かない（見どころ列挙はaside💡・出演時間予測・まとめと重複し、見出しの主題外のため2026-07-17に廃止）。
「トリの予測」「VTR企画のタイミング」を今回の情報で更新する。企画コーナーの詳細（候補曲リスト等）は出演者・歌唱曲一覧のカード側にのみ記載し、🔍カードは時間帯の傾向だけを書く。通常回/特番の構成説明（🕒）は共通のため原則維持。

**【SEO】** 「トリの予測」は断言調で書く（「〜の可能性があります」より「〜が最有力」）。検索者は答えを求めており、曖昧な表現は離脱につながる。根拠は主観形容詞ではなく「過去の放送での配置傾向」などの事実で短く添える。

### 8. H2：実際の出演順【放送後更新】
**放送前は直前回（資料②）のアーカイブデータを暫定掲載する。**
アーカイブパネル内の日付・出演者・タイムライン・REPORTを直前回の情報に差し替える。
放送後にリアルタイムで今回の実データに書き換える運用。

### 9. H2：見逃し配信・再放送はある？

**ブロック本体：スキップ（構造変更・テンプレートコピー不要）。** ドラフトの既存ブロックをそのまま維持する。
復元が必要な場合は `template_mste_static.txt` をReadする（バックアップ参照用）。

**YouTube埋め込みURL：毎回最新動画に差し替える（必須）。**

#### YouTube最新動画URL取得手順

1. WebFetch `https://www.youtube.com/feeds/videos.xml?channel_id=UCStFsybJM6SjDFvcacVcanw`（RSSフィード。チャンネルの`/videos`ページは動的レンダリングで中身が取得できないため使わない）
2. フィード先頭（最新）エントリの動画ID（`yt:videoId`）を確認する
3. 以下の2箇所を取得した動画IDに差し替える：

```
<!-- wp:embed {"url":"https://www.youtube.com/watch?v=【動画ID】", ...} -->
...
https://www.youtube.com/watch?v=【動画ID】
...
<!-- /wp:embed -->
```

> RSSフィードでも動画IDが取得できない場合は「YouTubeの動画IDが確認できませんでした。手動で差し替えてください」とユーザーに伝えて、テンプレートの既存URLのままにする。

### 10. H2：ミュージックステーション（Mステ）過去のタイムテーブル一覧
見出し文言は固定（「ミュージックステーション」「Mステ」「タイムテーブル」が共起する唯一のH2。書き換え・省略しない）。

**このセクションにH3フルカードは置かない**（直前回の詳細はアーカイブパネル（手順8）が正の置き場所。重複解消のため2026-07-17に構成変更）。過去回はすべて `<details>`（▼さらに過去のタイムテーブルを見る）内のコンパクトカード（`<h3>`日付見出し＋PAST DATA ARCHIVEヘッダー＋時間帯別リスト＋💡総評1行）で持つ。

| 操作 | 内容 |
|---|---|
| 追加 | 手順1③で退避した「上書き前のアーカイブパネル」のデータを、details先頭へコンパクト形式で追加（資料②のデータではない） |
| 削除 | details内で最も古い回を削除（直近3〜4回分を維持） |

アーカイブパネル→コンパクト形式への要約変換は自動で行う（時間帯・アーティスト・歌唱曲・ピーク表記・💡総評を保持する）。

**完了後セルフチェック**：過去一覧の新規追加カードの放送日と、アーカイブパネル（手順8）の放送日が同じになっていないかGrepで確認する（同じ日付が2箇所に出ていたら③と資料②の取り違え）。

### 11. H2：よくある質問（FAQ）
Q1〜Q4の質問・回答を今回の出演者に合わせて書き換える。**Q5は固定質問のため書き換えない**（CDTV記事のQ5と同型。エイリアス関係をGoogleと読者に伝える目的）。FAQの採用基準・枠の定式・回答原則は `rewrite_common_rules.md` **11章**（/blog-kaigi 2026-07-17決定）に従う。
- Q1: タイムテーブルの事前発表（定型。回答に放送枠「HH:MM〜HH:MM」を明記し「何時から何時まで」の検索意図を吸収する）
- Q2: 今回最大の目玉アーティストの出番予測
- Q3: 動員力最大のファンダムを持つグループの出番予測（例：SixTONESとSnow Man。「なぜ出演者が多いのか」のような編集側説明の質問は置かない）
- Q4: 見逃し配信の有無（定型・2文以内）
- Q5:（固定・編集不要）「Mステとは何の略ですか？」→「Mステは『ミュージックステーション』の略称です。テレビ朝日系列で毎週金曜よる8時から生放送されている音楽番組で、1986年の放送開始から続く長寿番組として知られています。」

**【SEO】** Q文は「読者が検索バーに実際に打ち込む言葉」で書く。「〜はありますか？」「〜は何時頃？」など口語的な疑問形が検索意図と一致しやすい。A文は**結論先出しの1〜2文**（結論→根拠）。出演者の全列挙など本文と重複する長い列挙は入れない。予測系の回答（Q2・Q3）には根拠（過去の傾向）と「実際の出演順は放送開始後に本ページで更新します」の再訪導線を添える。

### 12. H2：【まとめ】
放送日・出演者・見どころを今回の情報で書き換える。

**スキップ（Read・Edit不要）。ドラフトの既存ブロックをそのまま維持する。**
