## 作業方針（必読）

| 原則 | 内容 |
|---|---|
| **全体Readしない** | Grepで対象セクションの行番号を特定してからoffset/limitで必要範囲だけ読む |
| **WriteしないでEditする** | 変更箇所のみEditで差分更新。Write（全体上書き）は使用しない |
| **見出し順は固定** | 先頭H2は必ず「最新タイムテーブル【日付】」。FAQは「まとめ」の直前に固定（見出し順を並べ替えない） |
| **ショートコードは独立ブロック** | `[nopc][title]`・`[nopc][mokujimae]`・`[nopc][originalsc]` はそれぞれ独立した `<!-- wp:paragraph -->` ブロックに配置する |
| **静的セクションはスキップ** | ナビゲーション/アフィリエイトブロック（まとめH2直下のdiv）はRead・Edit不要 |
| **1段落1テーマ** | 「〇〇が発表された＋理由・状況説明」のような2クローズ以上の複合文は、生成時点で別々の`<!-- wp:paragraph -->`に分割する（ユーザーの手直し工数を減らす） |
| **継続性ワードは番組名とセット** | 「〇年連続」「引き続き」「再び」等は、対象が別番組（例：司会者の前番組）である場合、その番組名を同じ文・同じ要約（バッジ等）内に必ず明記する。今年が初回の特番でこの表現を使うと「本番組が去年も放送された」と誤読されるため |

## リライト手順

### 1. ファイル準備（放送後の初回対応時のみ）

放送後にアーカイブセクションを新設する場合のみ、STARの`template_star_static.txt`「過去のタイムテーブル」構造を参考に`<details>`ブロックを追加する。年1回特番のため複数年分のアーカイブ管理は2回目の放送以降に発生する。

---

### 2. タイトル・メタディスクリプション

**タイトル形式（evergreen優先）**
```
NHK夏の音楽祭 うたであえたら2026 タイムテーブル｜出演者や順番を随時更新
```
「司会3名」「出演者31組」のような**その時点限りの具体数値はタイトルに焼き込まない**。追加発表のたびに陳腐化・不整合が起きるため、「出演者や順番を随時更新」のようなevergreenな文言を基本形として維持する。年号は末尾の【2026】ブラケットではなく番組名に直接付ける（「うたであえたら2026」）。「タイムテーブル」を必ず含める。

**メタディスクリプション（100〜115字目安）**

放送日・番組名・最新の確定情報（司会/出演者の発表状況）を含める。「どこよりも早く」「最速で」等の誇張した速報訴求語は使わない（[[feedback-shira-title-meta-evergreen]]）。

**確定した文字列は手順11のJSON-LD descriptionに丸ごとコピーする（要約・言い換え不可）。メタの見出し「メタディスクリプション（XX字）：」の字数表記も、確定後の実文字数と必ず一致させる。**

---

### 3. aside①「番組情報まとめ」

Grepでaria-label="番組情報まとめ"を特定し、前後100行を読む。

| 要素 | 更新内容 |
|---|---|
| 放送日時 | 確定していれば時刻を反映（現状固定：8.15(土) 18:05-22:00） |
| 司会：行 | 発表済みの司会者名をすべて記載。未発表分は「他N名は未発表」等 |
| 出演者：行 | 発表済みアーティストを列挙。未発表の場合は現状維持 |
| リンクボタン | 公式サイトURLは固定 |

---

### 4. H2：最新タイムテーブル【日付】＋出演アーティスト・歌唱曲一覧

Grepでaria-label="タイムテーブル 前半"／"タイムテーブル 後半"／"出演者＆歌唱曲一覧"を特定し、それぞれ前後100行を読む。

| 状況 | 対応 |
|---|---|
| 出演順が一部でも判明 | 該当時間帯の「現在未発表です」を実データ（時間・アーティスト・曲）に差し替える |
| 出演アーティストのみ判明（順不同） | タイムテーブルは「未発表」のまま維持し、出演アーティスト・歌唱曲一覧カードのみ実データに更新 |
| 全て未発表 | 現状のプレースホルダー文言を維持（書き換え不要） |

出演アーティスト・歌唱曲一覧カードは、実データが入り次第STARの「アーティストラインナップ」形式（アーティスト名・曲名・一言コメントの3段組み）に作り直すことを検討する。**アーティスト名のみ発表・曲目が未発表の場合**は、各カードを「アーティスト名＋『曲目：未発表』」の2行構成にする（曲名を推測・捏造しない）。カードのフッター注記に発表基準日・追加発表予定の有無・公開収録不参加者などの補足情報をまとめる。

---

### 5. H2：司会は大森元貴（Mrs. GREEN APPLE）｜音楽番組初司会

Grepで「司会は大森元貴」を特定し、前後60行を読む。

残る2名の司会者が発表された場合、見出しを「司会は大森元貴・〇〇・〇〇の3名」等に更新し（｜以降の補足サブテキストは本文と重複するため省略可）、本文に新司会者の紹介を追加する。文章は1人1段落に分割する。

3名が確定した段階では、文章パートに加えて「MC PROFILE」スタイルのプロフィールカード（`aria-label="司会者プロフィール"`、既存の「番組の見どころ」HIGHLIGHTSカードと同系統のデザイン）を見出し直後に追加すると、テキストとコードのバランスが良くなる。各司会者を名前＋バッジ（肩書き・経歴の一言）＋コメント引用の3行構成にする。

**司会者の経歴紹介で「〇年連続」「引き続き」等の継続性ワードを使う場合、対象が別番組（前年の担当番組）であれば、その番組名を同じ文・同じバッジ内に必ず明記する。**年1回特番は「今年が初回」が記事の根幹事実のため、番組名を省略すると「本番組が去年も放送された」と誤読される。本文とプロフィールカードのバッジ両方に同じ配慮が必要（片方だけ直して他方を見落としやすい）。

### 11. 構造化データ（JSON-LD）

Grepで`MANUAL_JSONLD`を特定し、後続150行を読む。

| フィールド | 更新内容 |
|---|---|
| `headline` | 手順2で確定したタイトルと**一字一句一致** |
| `description` | 手順2で確定したメタディスクリプションと**一字一句一致** |
| `dateModified` | 更新日時（ISO形式） |
| `keywords` | 発表済み出演者名・司会名を反映 |
| BroadcastEvent `performer` | 発表済みの司会者・出演者（`Person`/`MusicGroup`型） |
| BroadcastEvent `startDate`/`endDate` | 放送時間が確定・変更された場合のみ |
| FAQPage `mainEntity` | 記事本文のQ&Aと**一字一句一致**（4問） |
| `image`/`thumbnailUrl` | アイキャッチ画像URLが提供された場合のみ`{画像URL}`から差し替え |

#### 固定箇所（変更不要）

`@id` / `url` / `datePublished` / `dateCreated` / `author` / `publisher` / `about` / `BreadcrumbList`

#### アーティストの型判定

| 区分 | 型 |
|---|---|
| グループ・バンド | `{ "@type": "MusicGroup", "name": "アーティスト名" }` |
| ソロ（個人名） | `{ "@type": "Person", "name": "アーティスト名" }` |

`name`は本文表記に括弧補足（例：「増田惠子（ピンク・レディー）」）があっても、JSON-LDでは括弧を外した本名・活動名のみを入れる（`"増田惠子"`）。司会者は本文で紹介済みでもBroadcastEventの`performer`配列への追加漏れが起きやすいので、出演アーティスト一覧を更新したら**司会者3名も含めて全員分**を数え直す。
