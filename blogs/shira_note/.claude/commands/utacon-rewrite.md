# うたコン タイムテーブル記事リライト

うたコン（NHK総合）のタイムテーブル記事を最新情報にリライトし、`draft_utacon.txt` として上書き保存する。

## 事前情報の確認

### 必須情報（資料①）
- 放送日（YYYY年MM月DD日）
- 放送時間・チャンネル（NHK総合）
- 今回のテーマ名（例：「夏うたで楽しまナイト！」。タイトル・メタには入れない。本文・aside・締め文には使用可）
- 出演アーティスト一覧
- 各アーティストの披露曲（発表されていれば）
- 今回の見どころ・注目ポイント（初コラボ・追悼企画・初披露など）

### 推奨情報（資料②）
- 直前回のX（旧Twitter）投稿分析データ
  - 各アーティストの出演時間帯・順番
  - SNS反応のピーク・盛り上がりポイント
  - 「調査結果」として過去一覧カードに反映する
  - **Xポストのタイムスタンプから再構成されたデータは、放送尺（19:57〜20:42・45分間）を超える時間帯が含まれていないか必ず確認する**。超過している場合はユーザーに確認し、「頃」表記を残しつつ最終ブロックを「放送終了頃」に丸めるなど、ユーザー承認の上で処理する（2026-07-15 うたコン新規記事立ち上げ時に7/14データで実際に発生）

### その他
- アイキャッチ画像URL（構造化データ（JSON-LD）に反映）。**未確定の間は`draft_utacon.txt`内の`utacon-timetable-PLACEHOLDER.webp`のままにし、画像URL確定後に一括Grep置換する**

### 資料①が「出演者未発表」の場合

次回放送の出演者がまだ発表されていない状態でリライトを依頼された場合、以下の方針で進める。

| セクション | 扱い |
|---|---|
| aside・最新タイムテーブル・出演者一覧・出演時間予測・順番の傾向 | 「出演アーティスト未発表・判明次第更新」のプレースホルダー文言に差し替える（個別アーティストカードは削除し、通知ブロック1枠にする） |
| 実際の出演順アーカイブパネル（手順8） | 資料②（直前回データ）をそのまま「直前回」として掲載する（通常運用と同じ） |
| 過去のタイムテーブル一覧（手順10） | 直前回を新規フルカードとして追加する処理は通常通り実施 |
| FAQ・まとめ・JSON-LD | 出演者名を含む断定的な記述を避け、「時点で未発表・判明次第追記」の文言に統一。JSON-LD の `mentions`/`performer`/`itemListElement` は空配列にする |

出演者が発表され次第、再度 `/utacon-rewrite` を実行し、このプレースホルダー状態から通常のフル更新に戻す。

### 歌唱曲・出演順の事前発表は「通常回か特番か」で決まる（全番組共通の前提・2026-07-15確定）

**歌唱曲の事前発表そのものはどの番組でも起こりうる**。うたコンが特別に非公開というわけではない。判断基準は「特番・スペシャル回か、通常回か」：

| 放送形態 | 歌唱曲・出演順の事前発表 |
|---|---|
| 特番・スペシャル回（音楽の日・THE MUSIC DAY等） | 事前に一部発表されることがある |
| 通常回（うたコンの毎週放送はこちらが基本） | **基本的に事前発表されない** |

うたコンの通常回をリライトする場合、出演者のみ判明した状態で立ち上げるのが基本運用であり、「◯月◯日に歌唱曲が発表される」という特定日付を本文・aside・FAQ・JSON-LDのどこにも書かない（2026-07-21放送分の立ち上げ時、実際には存在しない「7/17発表予定」という情報を書いてしまい、6箇所以上の手戻りが発生した）。**資料①受領時に「今回は通常回か特番か」を必ず確認する。**

| セクション | 扱い（通常回の場合） |
|---|---|
| 最新タイムテーブル（手順4） | 「歌唱曲・出演順は基本的に事前発表なし。放送に合わせて更新」の通知ブロックのみ表示（時間帯カードは作らない） |
| aside STATUS/TIMETABLE | 「歌唱曲・出演順は基本的に事前発表なし。判明次第更新」で統一 |
| 出演者一覧（手順5） | 出演者カードは表示するが、各カードの説明文は「歌唱曲は現時点で未発表。」で統一する（曲名・発表日を創作しない） |
| FAQ Q1 | 出演者は判明済み・歌唱曲/出演順は基本的に事前発表がない旨を明記。「事前に情報が公開された場合や放送開始後は、本ページをリアルタイムで更新します」で締める |

**特番・スペシャル回、または事前に歌唱曲の一部が公開された場合**：判明した範囲で時間帯カード・アーティストカードを更新し、日付を明記してよい（推測で日付を先取りしない）。
歌唱曲・出演順が判明次第、再度 `/utacon-rewrite` を実行してタイムテーブル・出演者カードをフル更新する。

## リライト手順

### 1. 直前回ドラフトの「過去のタイムテーブル一覧」を Grep → Read で抽出

```
Grep パターン: "過去のタイムテーブル一覧"
  → ヒット行番号を確認し、Read offset=<その行> limit=250 で抽出
```
取得目的：H3フルカードを次回ドラフトに反映するため。

### 1'. 上書き前の「実際の出演順【放送後更新】」アーカイブパネルを Grep → Read で抽出（退避データ）

```
Grep パターン: "実際の出演順"
  → ヒット行番号を確認し、Read offset=<その行> limit=100 で抽出
```
取得目的：このアーカイブパネルの中身は**今回の資料②ではなく、今回の資料②で上書きされる「前々回」のデータ**。手順10で過去一覧のH3フルカードに追加するのはこの1'のデータであり、資料②（今回アーカイブパネルに新しく入れるデータ）ではない。**1'と資料②を取り違えないこと**（2026-07-18 Mステの同種ミスを受けて全番組共通で明文化。過去一覧に資料②の日付を入れると、実際の出演順アーカイブパネルと内容が重複する）。

### 2. タイトル・メタディスクリプション

**タイトル**
```
うたコン タイムテーブル【M月DD日】出演者・出演順をリアルタイム更新
```
- 「うたコン タイムテーブル」を冒頭に固定
- 放送日を【】で表記
- テーマ名・具体数値は入れない（`rewrite_common_rules.md` 3章）
- 「最速」等の誇張語は使わない

**メタディスクリプション（120字以内）**

```
<!-- wp:paragraph -->
タイトル：うたコン タイムテーブル【M月DD日】出演者・出演順をリアルタイム更新
メタディスクリプション（XXX字）：
うたコン（M月DD日・NHK総合）タイムテーブルを速報更新！{最大注目ポイント1}・{注目ポイント2}ら{人数}組出演。出演順・登場時間の目安を随時追記。
<!-- /wp:paragraph -->
```

> 文字数表記（XXX字）は実際の文字数と必ず一致させる。複数候補を提示する場合は提示前にPowerShellの`.Length`で一括実測する（`rewrite_common_rules.md` 9章）。
> JSON-LDの`description`もメタディスクリプションと一字一句一致させる（手順14で最終確認）。

### 3. 冒頭段落・aside カード

| プレースホルダー | 内容 |
|---|---|
| 放送日時 | YYYY年MM月DD日(曜) HH:MM〜HH:MM（NHK総合） |
| STATUS | 「出演アーティスト解禁」等現時点の状態（歌唱曲未発表なら「歌唱曲・出演順は◯月◯日発表予定」も明記） |
| ARTISTS | 出演者を `/` 区切りで列挙 |
| TIMETABLE（進行イメージ） | 放送時間・進行の流れが伝わる1〜2文 |
| 編集者コメント | 今回のテーマ名・特記事項（コラボ・追悼企画等） |
| 最終更新日 | 記事作成日（YYYY.MM.DD） |
| 「最新情報を確認する」リンク | href（標準値 `#st-toc-h-1`）は本文最初のH2見出しへのTOCジャンプ用 |

締め文（`[nopc][mokujimae][/nopc]` 直後の段落）は今回の見どころに合わせて更新する。主観形容詞（「豪華」等）は使わない。

**【SEO】** 締め文は「推しの出番を逃したくない」という検索者の切実な動機（ペルソナ：佐藤真由）に直接応えるコピーにする。

### 4. H2：うたコン最新タイムテーブル
歌唱曲・出演順が発表されている場合は時間帯カード形式に更新する（Mステ同型のタイムテーブルカード）。
未発表の場合は「未発表」通知ブロックのまま、発表予定日のみ更新する。

### 5. H2：うたコンの出演者一覧
各アーティストカードを最新の出演者に差し替える。歌唱曲が判明していれば「「曲名」を披露。」に更新、未発表なら「歌唱曲は◯月◯日発表予定。」のまま。
`ARTIST LINEUP (YYYY.MM.DD)` の日付も更新する。

### 6. H2：うたコンの出演時間は何時頃？【目安】
見出しの「うたコンの」は固定（検索順位強化のための固定表記。変更しない）。
3ブロック（序盤・中盤・終盤）の説明を直前回の傾向＋今回の出演者に合わせて書き換える。
アーカイブの蓄積が少ない間（1〜2回分）は、断定調を避け「前回は〜でした」という事実ベースの記述に留める。3回分以上蓄積したら断定調（「〜が有力」）への切り替えを検討する。

### 7. H2：うたコンの順番の傾向
カード内の「テーマ企画の存在」「コラボ企画」「公式SNSの連動」を今回の情報で更新する。

### 8. H2：うたコンの実際の出演順【放送後更新】
**放送前は直前回（資料②）のアーカイブデータを暫定掲載する。**
アーカイブパネル内の日付・出演者・タイムライン・REPORTを直前回の情報に差し替える。
放送後にリアルタイムで今回の実データに書き換える運用。

### 9. H2：うたコンの見逃し配信・再放送はある？
**うたコンは放送終了後から1週間限定で、NHKの公式配信サービス「NHK ONE」にて見逃し配信される（2026-07-15確定）。** 「配信の可能性あり」等の曖昧な表現ではなく、「あります」で断定してよい。配信期間（1週間限定）を明記する。**「NHKプラス」ではなく「NHK ONE」が正式名称のため誤記に注意**。

### 10. H2：うたコン過去のタイムテーブル一覧
見出し文言は固定。以下のルールで更新する。

| 操作 | 内容 |
|---|---|
| 追加 | 手順1'で退避した「上書き前の実際の出演順アーカイブパネル」のデータをフルカード形式でH3として先頭に追加（資料②のデータではない） |
| 維持 | 2回前の回はH3フルカードのまま維持 |
| 移動 | 3回前の回を `<details>` 折りたたみブロックの先頭に移動（コンパクト形式に変換） |
| 削除 | `<details>` 内で最も古い回を削除 |

**完了後セルフチェック**：過去一覧の新規追加カードの放送日と、実際の出演順アーカイブパネル（手順8）の放送日が同じになっていないかGrepで確認する（同じ日付が2箇所に出ていたら1'と資料②の取り違え）。

新規フルカードの根拠注記（フッター）は以下の定型文で統一する。
```
※Xのリアルタイム投稿を参考にまとめた推定タイムラインです。（不明点があれば「◯◯の歌唱曲、正確な放送終了時刻は確認できていません」等を追記）
```

### 11. H2：よくある質問（FAQ）
Q1・Q4を今回の内容で更新する。**Q2・Q3・Q5は固定質問のため書き換えない**。
- Q1: 今回の歌唱曲・出演順の事前発表有無（今回の出演者名を反映）
- Q2:（固定）「うたコンとはどんな番組ですか？」
- Q3:（固定）「見逃し配信はありますか？」
- Q4: 前回放送の内容サマリー（直前回データに更新）
- Q5:（固定）「うたコンの放送時間は何時からですか？」→ 放送日時のみ更新

**【SEO】** Q文は「読者が検索バーに実際に打ち込む言葉」で書く。A文は冒頭の1文で結論を出し、補足説明を続ける構造にする。

### 12. H2：【まとめ】
放送日・出演者・見どころを今回の情報で書き換える。

### 13. ナビゲーション・アフィリエイトブロック（固定）

### 14. 構造化データ（JSON-LD）の更新

#### 差し替え箇所

| 項目 | 更新内容 |
|---|---|
| `headline` | タイトル（WordPressタイトル欄と同じ文字列） |
| `description` | メタディスクリプションと同じ文字列 |
| `dateModified` | 記事作成・更新日時（ISO形式：YYYY-MM-DDTHH:MM:SS+09:00） |
| `image` / `thumbnailUrl` | 今回のアイキャッチ画像URLに更新（未確定ならPLACEHOLDERのまま） |
| `mentions` / `performer` | 出演者をMusicGroup/Person型のJSON配列で列挙 |
| BroadcastEvent `name` | 「うたコン（YYYY年MM月DD日放送・{テーマ名}）」に更新 |
| BroadcastEvent `description` | 今回の放送説明文に更新 |
| BroadcastEvent `startDate` / `endDate` | 放送日時（ISO形式）に更新 |
| ItemList `name` | 「YYYY年MM月DD日 うたコン出演者一覧」に更新 |
| ItemList `numberOfItems` | 出演者数に更新 |
| ItemList `itemListElement` | 出演者を最新に更新 |
| FAQPage `mainEntity` | 記事本文のQ&Aと**一字一句一致**させる（Q1・Q4は今回の内容、Q2・Q3・Q5は固定文のまま維持。計5問） |
| `keywords` 配列 | 固定6語：`"うたコン タイムテーブル", "うたコン 出演者", "うたコン 順番", "うたコン 今日", "うたコン {年}", "うたコン {月日}"`。放送日部分のみ更新し、他は書き換えない |

#### 固定箇所（変更不要）

- `datePublished`：新規記事作成日時で固定（以後変更しない）
- `author` / `publisher`
- `about`（うたコン番組情報）
- `BreadcrumbList`（うたコンのWPカテゴリURLが確定していない場合はプレースホルダーのまま。確定後に一括置換）

#### アーティストの型判定

| 区分 | 型 |
|---|---|
| グループ・バンド | `{ "@type": "MusicGroup", "name": "アーティスト名" }` |
| ソロ（個人名） | `{ "@type": "Person", "name": "アーティスト名" }` |

判定は`generate_jsonld.py`のルール（`&`/`group`含む・英字カタカナ→グループ、漢字ソロ名義→個人）に準拠。迷う場合は`tools/generate_jsonld.py`を使う。

#### 注意事項

- アイキャッチ画像URLは事前確認済みのものを使用する（未確定ならPLACEHOLDERを維持しユーザーに催促）
- FAQのQ&A文は記事本文と**一字一句一致**させる
