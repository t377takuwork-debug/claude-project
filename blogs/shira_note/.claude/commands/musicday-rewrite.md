### 2. タイトル・メタディスクリプション

**タイトル**
```
THE MUSIC DAY YYYY タイムテーブル＆出演順番は？最新情報まとめ
```
- 年号（YYYY）を更新。「THE MUSIC DAY」を冒頭に固定

**メタディスクリプション（120字以内）**

冒頭を「「THE MUSIC DAY YYYY」M月DD日放送の最新情報！」で始め、今年最大の注目ポイント（話題アーティスト・企画）とタイムテーブル速報性を含めて自由に構成する。

| 項目 | 内容 |
|---|---|
| 冒頭 | 「THE MUSIC DAY YYYY」M月DD日放送の最新情報！（年号・日付を更新） |
| 注目ポイント | 話題アーティスト・テーマ・シャッフルメドレーなど今年固有の見どころを優先 |
| 文字数 | 90〜115字を目安。120字を超えない |

### 5. H2：タイムテーブルと出演順番

**年が変わる場合のみ** H2テキストの年号（例：`【2026年】`→`【2027年】`）を更新する。同年内の更新はスキップ。

H2直下のリード文は出演組数・発表状況に合わせて書き換える。

**前半ブロック（15〜18時台）・後半ブロック（19〜22時台）の更新**

| 要素 | タイムテーブル未発表時 | タイムテーブル発表後 |
|---|---|---|
| 各時間帯説明文（p） | 現状のまま維持 | 今年固有の傾向・出演者に合わせて書き換える |
| 各時間帯エントリ（時間・アーティスト） | `未発表（〇〇）` のまま維持 | 判明した出演者・楽曲を記載 |

タイムテーブル解禁前は時間帯カードの**説明文のみ**を今年の顔ぶれに合わせて更新し、エントリ部分は「未発表」のまま保持する。

**`[nopc][originalsc][/nopc]`** は15〜18時台カードと19〜22時台カードの間に独立ブロックとして維持する（移動・削除しない）。

時間帯説明文は「今年の出演者名を根拠にした具体的な予測」として書く。汎用文（「若手が登場する傾向があります」等）は使わない。

**このセクションは年をまたいで情報が蓄積される。** 過去実績を残しつつ、今年の予測を更新する。

| 今年の予想時期 | 過去傾向から今年の予測を更新 |
| CHECK!ボックス | 今年の予測解禁タイミングを更新 |

**タイムテーブル解禁後は**「XXXX年の解禁タイミング（実績）」として過去実績に変換する。

### 13. 構造化データ（JSON-LD）の更新

JSON-LDは常にファイル末尾の固定ブロック。更新フィールドの値はcontextから直接構築できるため、**ReadなしでEditを実行してよい**（Grepで行番号を得たらそのままEditへ進む）。

#### 毎回更新する箇所

| フィールド | 更新内容 |
|---|---|
| `description` | メタディスクリプションと同じ文字列 |
| `dateModified` | 更新日（ISO形式：YYYY-MM-DDTHH:MM:SS+09:00） |
| `keywords` | 今年の出演者名・テーマなどを更新 |
| `mentions` | 司会者・出演者（MusicGroup/Person型）を更新 |
| BroadcastEvent `description` | 今年の主な見どころを含む放送説明文 |
| FAQPage `mainEntity` | 記事本文のQ&Aと**一字一句一致**させる（4問） |

#### 変更がある場合のみ更新する箇所

| フィールド | 更新タイミング |
|---|---|
| `headline` | タイトルを変更した場合 |
| `image` / `thumbnailUrl` | 新しいアイキャッチ画像URLが提供された場合 |
| BroadcastEvent `name` | 年号が変わる場合 |
| BroadcastEvent `startDate` / `endDate` | 放送日時が確定・変更された場合 |
| BroadcastEvent `location` | 会場が変更された場合 |

#### 固定箇所（変更不要）

- `datePublished` / `dateCreated`：2026-04-27T00:00:00+09:00（固定）
- `@id` / `url`（記事URL）
- `author` / `publisher`
- `about`（THE MUSIC DAY番組情報）

#### アーティストの型判定

| 区分 | 型 |
|---|---|
| グループ・バンド | `{ "@type": "MusicGroup", "name": "アーティスト名" }` |
| ソロ（個人名） | `{ "@type": "Person", "name": "アーティスト名" }` |

#### 注意事項

- FAQのJSON-LDは手順10のFAQ本文を書いた後に記事テキストから合わせる
- ユーザーが本文を修正した場合も必ず再確認する
