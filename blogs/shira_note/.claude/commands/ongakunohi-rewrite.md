## 作業方針（必読）

| 原則 | 内容 |
|---|---|
| **WriteしないでEditする** | 変更箇所のみEditで差分更新。Write（全体上書き）は使用しない |
| **静的セクションはスキップ** | ナビゲーション/アフィリエイトブロック（まとめH2直下のdiv）はRead・Edit不要 |
| **ショートコードは独立ブロック** | `[nopc][title]`・`[nopc][mokujimae]`・`[nopc][originalsc]` はそれぞれ独立した `<!-- wp:paragraph -->` ブロックに配置する |
| **全体Readしない** | Grepで対象セクションの行番号を特定してからoffset/limitで必要範囲だけ読む |
| **独立セクションは並行Edit** | old_stringが互いに重複しない場合は1メッセージに複数Editを束ねる |
| **段落はシンプル・分割** | 1段落＝1テーマ。冒頭リード文・H2直後リード文は1文に絞る。複数情報は別 `<!-- wp:paragraph -->` ブロックに分ける |

## リライト手順

### 1. タイトル・メタディスクリプション・冒頭リード文

**タイトル**
```
【YYYY】音楽の日タイムテーブル｜出演者の順番や出演時間まとめ(随時更新)
```
- 年号（YYYY）を更新。「【YYYY】音楽の日」を冒頭に固定

**冒頭リード文**（`[nopc][title]`直前の段落、放送回数テキスト）

以下の固定テンプレートを使う（1〜2文のみ）：
```
TBS系の夏の大型音楽特番「音楽の日YYYY」の放送が決定！

YYYY年で第XX回目の放送を迎えます。
```

**メタディスクリプション（120字以内）**

冒頭を「「音楽の日YYYY」M月DD日生放送！」で始め、今年最大の注目ポイントとタイムテーブル速報性を含めて自由に構成する。

| 項目 | 内容 |
|---|---|
| 冒頭 | 「音楽の日YYYY」M月DD日生放送！（年号・日付を更新） |
| 注目ポイント | 話題アーティスト・テーマなど今年固有の見どころを優先 |
| 文字数 | 90〜115字を目安。120字を超えない |

### 3. 締め文（`[nopc][mokujimae][/nopc]` 直後の段落）

**放送後：**

```
出演者・歌唱曲・タイムテーブル情報を更新済みです。ぜひブックマークして次回の放送にも備えてください。
```

### 4. H2：タイムテーブル

H2テキストの年号を更新する。

H2直下のリード文は以下の固定テンプレートを使う（毎回自由生成しない）：
```
「音楽の日YYYY」のタイムテーブルは、例年放送前日に発表される傾向にあります。
```

**タイムテーブルカードの更新**

| 要素 | タイムテーブル未発表時 | タイムテーブル発表後 |
|---|---|---|
| ヘッダー年号 | `音楽の日 YYYY` に更新 | 同左 |
| ステータスバッジ | `COMING SOON` のまま維持 | `LIVE` または削除 |
| 各時間帯エントリ（時間・テキスト） | 「出演アーティスト・歌唱曲の発表をお待ちください」等 | 判明した出演者・楽曲を記載 |
| フッター注記 | 現状維持 | 放送日に合わせて更新 |

タイムテーブル未発表時は時間帯の**説明文のみ**今年の顔ぶれ・発表状況に合わせて更新し、エントリ枠は維持する。

**`[nopc][originalsc][/nopc]`** はタイムテーブルカードと次セクションの間に独立ブロックとして維持する。

カード後の段落テキスト（「現時点では〜」）も発表状況に応じて更新する。

### 5. H2：出演者グリッド

H2テキストの年号を更新する。

**ARTISTSグリッド（2カラム）**を最新の出演者に更新する。

| 操作 | 内容 |
|---|---|
| ヘッダーの発表弾表示 | `ARTISTS (第X弾発表)` を更新 |

| フッターの注記テキスト | 発表状況・続報予定に合わせて更新 |

**PROJECTSカード**を今年の企画に更新する。

| 操作 | 内容 |
|---|---|
| 各カードのタイトル・説明文 | 今年の企画内容に差し替え |
| 不要なカード | 削除可（今年対応しない企画は除去） |
| 新規企画 | カードを追加してよい |

企画情報が未発表の場合はPROJECTSエントリを「詳細発表予定」として維持する。

グリッド後の段落テキストも発表状況に応じて更新する。

**`[nopc][originalsc][/nopc]`** は発表スケジュールカードの後に独立ブロックとして維持する。

### 12. 構造化データ（JSON-LD）の更新

JSON-LDは常にファイル末尾の固定ブロック。構造は以下の通り（この順序を崩さない）：

```
<!-- wp:html -->
<!-- MANUAL_JSONLD -->
<script type="application/ld+json">
...
</script>
<!-- /wp:html -->
```

`<!-- MANUAL_JSONLD -->` マーカーは `<!-- wp:html -->` の内側に配置する。更新フィールドの値はcontextから直接構築できるため、**ReadなしでEditを実行してよい**（Grepで行番号を得たらそのままEditへ進む）。

#### 毎回更新する箇所

| フィールド | 更新内容 |
|---|---|
| `description`（BlogPosting） | メタディスクリプションと同じ文字列 |
| `dateModified` | 更新日（ISO形式：YYYY-MM-DDTHH:MM:SS+09:00） |
| `keywords` | 今年の出演者名・テーマなどを更新 |
| `mentions` | 出演者（MusicGroup/Person型）を更新 |
| BroadcastEvent `description` | 今年の見どころを含む放送説明文 |
| FAQPage `mainEntity` | 記事本文のQ&Aと**一字一句一致**させる（3問） |

#### 変更がある場合のみ更新する箇所

| フィールド | 更新タイミング |
|---|---|
| `headline` | タイトルを変更した場合 |
| ImageObject `url` | 新しいアイキャッチ画像URLが提供された場合 |
| BroadcastEvent `name` | 年号が変わる場合（例：`音楽の日YYYY`） |
| BroadcastEvent `startDate` / `endDate` | 放送日時が確定・変更された場合 |
| BroadcastEvent `location.name` | 会場が変更された場合 |
| ItemList `itemListElement` | タイムテーブルが確定した場合 |

#### 固定箇所（変更不要）

- `datePublished` / `dateCreated`：2026-04-27T00:00:00+09:00（固定）
- `@id` / `url`（記事URL）
- `author` / `publisher`
- `about`（音楽の日番組情報）

#### アーティストの型判定

| 区分 | 型 |
|---|---|
| グループ・バンド | `{ "@type": "MusicGroup", "name": "アーティスト名" }` |
| ソロ（個人名） | `{ "@type": "Person", "name": "アーティスト名" }` |

#### 注意事項

- FAQのJSON-LDは手順9のFAQ本文を書いた後に記事テキストから合わせる
- ユーザーが本文を修正した場合も必ず再確認する
