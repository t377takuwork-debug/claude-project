# shira_note 新規番組タイムテーブル記事の立ち上げ

既存のリライト対象外（コマンド一覧に無い）音楽番組のタイムテーブル記事を、一から作成する際の標準手順。
「新しい番組の記事を作りたい」と言われたら、このコマンドの手順に従う。

## 目的

`draft_utadeaetara.txt` 作成時（2026-07-01）に発生した手戻りを踏まえ、以下を1回のフローに統合する。

- 構成方針・URLスラッグなどの確認を1回のAskUserQuestionラウンドで済ませる
- 既存draft（同格の番組）の見出し順を必ず先に確認してから書く
- aria-label・専用リライトコマンド・CLAUDE.md登録を、記事本体の作成と**同時に**行う
- JSON-LD同期チェックを最終手順として必ず組み込む

---

## 手順

### 1. スコープ確認（1ラウンドで済ませる）

以下をまとめてAskUserQuestionまたはヒアリングで確認する。個別に何度も聞き返さない。

| 確認項目 | 内容 |
|---|---|
| 既存記事の有無 | ブログに既存記事があるか（あればリライト対象、無ければ新規） |
| 資料①の有無 | 放送日・出演者・見どころ等（CLAUDE.mdの捏造禁止ルールに従い必須） |
| テンプレート方針 | 既存のタイムテーブル記事と同じ型を踏襲するか |
| タイトル案 | 番組正式名称を含めた3案から選定 |
| URLスラッグ | 記事の固定リンク |

### 2. 同格の既存draftを1本読む（必須・最初に行う）

書き始める前に、**同じカテゴリ**（年1回特番なら`draft_star.txt`や`draft_musicday.txt`等）を1本読み、以下を確認する。

- H2の出現順（先頭＝最新タイムテーブル、FAQ＝まとめ直前、が共通ルール）
- 各セクションの`aria-label`命名パターン
- JSON-LDの`@graph`構成（BlogPosting/BroadcastEvent/FAQPage/BreadcrumbList）

出演者・タイムテーブルが未発表の段階でも、見出し自体は「最新タイムテーブル【日付】」を先頭に置き、中身だけ「未発表」にする（[[feedback_shira_new_article_structure]]）。

### 3. 記事本体を作成する

手順2で確認した見出し順・デザインパターンに合わせて`draft_{番組略称}.txt`を作成する。

**このとき、以下を最初から満たす（後付けにしない）：**

- すべてのカード状要素（`<aside>` / カスタム`<div>`）に `role="region" aria-label="..."` を付与する
- ショートコード（`[nopc][title]` 等）は独立した `<!-- wp:paragraph -->` ブロックに配置する
- JSON-LDの`headline`/`description`は、記事冒頭のタイトル・メタと一字一句一致させる

### 4. 専用リライトコマンドを同時に作成する

`.claude/commands/{番組略称}-rewrite.md` を、手順2で参照した既存コマンド（`star-rewrite.md`等）を土台に新規作成する。最低限、以下を含める。

- 実行フロー判定表（状況ごとにどの手順を実行するか）
- セクション特定用Grepマーカー一覧（手順3で付与したaria-labelと対応させる）
- JSON-LD更新フィールド一覧・FAQ整合性チェック
- スマートクォート確認コマンド

### 5. CLAUDE.md に登録する

`shira_note/CLAUDE.md`の以下2箇所に追記する（記事作成と同じセッション内で行う。後回しにしない）。

- 「スラッシュコマンド一覧」表
- 「ドラフトファイルの命名規則」表

### 6. 最終チェック

- スマートクォート置換（PowerShellコマンド実行）
- JSON-LD ⇔ 本文の整合性確認（headline / description / FAQ mainEntity）

---

## 補足：ユーザーが本文を手直しした後の対応

ユーザーがdraftファイルを直接編集した場合、**次のやり取りで必ず**以下を確認する（聞かれる前に自分から言う）。

- タイトル・メタディスクリプションが変わっていないか → 変わっていればJSON-LD `headline`/`description`を再同期
- FAQの質問・回答文言が変わっていないか → 変わっていればJSON-LD `FAQPage.mainEntity`を再同期
- 見出し構成（H2の順番）が変わっていないか → 変わっていれば[[feedback_shira_new_article_structure]]のルールと整合するか確認

これを黙って見逃すと、JSON-LDと本文が食い違ったまま公開されるリスクがある。
