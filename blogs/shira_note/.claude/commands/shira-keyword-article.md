# shira_note キーワード起点 新規記事制作

特定の狙いキーワード（例：「音楽の日 DREAMダンス」）から着手する、テーマ特化型SEO記事の制作フロー。
番組タイムテーブル速報draft（`draft_XXXX.txt`）とは別枠の、独立した新規記事として扱う。

「〇〇というキーワードで記事を作りたい」「このキーワードで上位表示を狙いたい」と言われたら、このコマンドの手順に従う。

## 位置づけ

- 対象は**キーワード起点の単発テーマ記事**。番組タイムテーブル速報（先頭H2＝最新タイムテーブル固定）とは構成原則が異なる。
- SEO判断のマスターガイドは `blogs/seo/SEO_guide.txt`。本コマンドの各ステップはこのガイドの原則（結論→詳細→補足、表・リスト活用、タイトル冒頭にベネフィット）に従う。
- 捏造禁止ルール（プロジェクト全体CLAUDE.md）は本フローにも適用する。事実情報（出演者・楽曲・演出内容等）が不明な場合は、ユーザーに資料提供を求める。憶測で断定しない。

---

## 手順

### Step 0. 資料インプット確認フェーズ（Step 1より前に必ず実施）

**WebSearchによる自動リサーチは、検索結果の要約スニペットしか取得できず、公式サイトの詳細ページ本文（事務所別メンバーの個別名など粒度の細かい情報）まではほぼ拾えない。** この精度限界を前提に、Step 1でWebSearchを走らせる前に、必ず以下をユーザーに確認する。

1. 「この記事に使える一次資料（公式サイト・公式SNS・プレスリリース・過去のリサーチメモなど）はお持ちですか？あれば先にいただけると、リサーチの手戻りを防げます」と聞く
2. 資料が提供された場合、それを資料①として確定情報の基盤にする（WebSearchは資料の裏取り・関連キーワード調査・検索意図調査のみに使う）
3. 資料が無い場合のみ、Step 1のWebSearchで一次情報の収集を試みる。ただし個別名・詳細スペック等の粒度が必要な企画（出演者個別名、事務所別チーム分けなど）は、WebSearchでは限界がある旨をユーザーに伝え、判明した範囲で一旦提示したうえで「情報の抜けがあれば追加で資料をいただきたい」と申し出る
4. 記事執筆の途中（Step 4・5）でユーザーが独自調査した資料・公式サイトの原文・画像等を追加提供した場合は、その時点で本文を資料に基づいて更新する（`draft_dancebattle.txt`制作時の実例を参照）

### Step 1. キーワード受付とキーワード分析

ユーザーから狙いキーワードを1つ受け取ったら、WebSearchで以下を調査する（Step 0で資料①が提供済みの場合は、事実情報は資料①を優先し、WebSearchは関連キーワード・検索意図調査に限定する）。

- 当該キーワードのGoogle検索結果（上位記事のタイトル・見出し傾向）
- 関連記事・SNS上の話題（何が語られているか、いつの情報か）
- 関連キーワード・共起語（検索結果のタイトル・スニペットから抽出）
- 想定される派生・関連検索キーワード（サジェスト相当をAIが推論）

**毎年恒例の企画・コーナーが対象の場合は、名称が年によって変わっていないか必ず確認する**（例：「音楽の日」のダンス企画は2023年開始→年により「ダンスバトル」「DREAM DANCE」と呼称が揺れている）。過去の呼称の検索需要が現行の公式名称より大きい場合があるため、両方の検索ボリューム感を比較し、通称と正式名称のどちらを主軸にするかをStep 2〜3で確定する。

自社サイト（shira-treat.com）に同一企画の過去記事が既に存在しないかも確認する。存在する場合は、そのURLスラッグ命名パターン・見出し構成を踏襲することを優先する（Step 5参照）。

調査結果は以下の形でユーザーに提示する。

| 区分 | 内容 |
|---|---|
| 上位記事の傾向 | タイトル・扱っている切り口 |
| 関連キーワード | キーワードaaa, bbb, ccc... |
| 共起語 | よく一緒に使われる語 |
| 想定検索キーワード（派生） | このキーワードで検索する人が併せて調べていそうな語 |
| 情報の鮮度 | いつの情報が最新か（放送前/放送後/アーカイブ化後 等） |

### Step 2. 検索意図・ペルソナの確認

Step 1の調査結果をもとに、以下をAIが仮説として提示し、AskUserQuestionで確認・微調整する（1ラウンドで済ませる）。

- 検索意図の分類（例：見逃し配信を探している／出演者情報を確認したい／感想・考察を探している）
- 具体ペルソナ（`persona_timetable.txt` の形式を踏襲。年齢層・視聴シーン・知りたいことの核）
- この記事で「最後に知りたいこと」＝記事冒頭に置く結論

事実情報（誰が出演した、何を歌った・演じた等）が確認できていない場合は、ここで資料提供をユーザーに依頼する（捏造禁止）。

### Step 3. タイトル設計（2段階プロセス）

SEO_guide.txtのタイトル設計原則（ベネフィット・鮮度・正確性を冒頭に）に加え、以下の2段階プロセスで進める。**分析フェーズでタイトル案を出力しない**（ユーザーのOKを待たずに先走らない）。

#### 3-A. 分析フェーズ（タイトル出力前）

以下を箇条書きで提示し、ユーザーの確認を取る。

1. **キーワード拡張整理**：メイン/サブキーワードから関連キーワード・共起語を抽出し、各サブキーワードが「どの文脈で自然に使えそうか」を一言添える（Step 1の調査結果を流用可）
2. **想定ペルソナ（2パターン）**：年齢層・興味関心・読む動機・特に価値を感じる情報・抱えやすい不安/疑問・流入経路（検索/SNS/Discover等）を含める
3. **検索意図（1つに絞る）**：今回のタイトルで狙う検索意図を1つ提示。未確定情報は「現時点で分かっている範囲」と明示する
4. 最後に必ず次の一文を付ける：「上記の【関連キーワード・共起語】【ペルソナ】【検索意図】で問題なければ「OK」と返信してください。修正があれば上書き指示をください。タイトル案はまだ出力しません。」

#### 3-B. タイトル出力フェーズ（OK後に実行）

以下のルールで**10本**生成する。

**共通ルール**
- メインキーワードは完全一致で全案に1回以上含める
- 各サブキーワードは最低1案に自然な文脈で含める（使えなかった場合は理由を一言補足）
- 記号（？｜、…等）は意味が自然な範囲で活用し、10本中4本以上で使用する（装飾目的のみの多用は禁止）
- タイトル長は32〜40文字前後を目安にする
- 主観的感想・スラング・過剰な煽り・「やさしく解説」等の上から目線表現は禁止。高校生でも直感的に意味が分かる自然な日本語にする
- 整理済み素材（Step 1・資料①）にない事実は追加しない。未発表情報は断定しない
- 文頭・文末の型が単調にならないようにする

**出力形式（10パターン）**
1. 汎用型（SEO重視） 2. 特化型（CTR重視） 3. 質問型（AI検索向け） 4. 速報性強調型 5. 総まとめ型 6. Discover向け（感情フックやや強め） 7. 企画内容推し型 8. 出演者ファン向け特化型 9. ライト層向け（読みやすさ重視） 10. ニッチ検索拾い型

**タイトル出力後、必ずキーワード使用チェックを付記する**（例：メインキーワード「〇〇」使用→1,2,3... サブキーワード「△△」使用→1,4,7...）。

### Step 4. 見出し構成（H2/H3）設計

SEO_guide.txtの構成原則（概要→詳細→補足、3つ以上の事象はテーブル/リスト化）に従い、以下4点をセットで提示して確認する（本文はまだ書かない）。

1. **構造化アウトライン（H2・H3のみ）**
2. **メタディスクリプション案**（70〜120字）
3. **スラッグ案**（ハイフン区切り最大3要素、英数字小文字のみ、1案）
4. **見出し設計の意図メモ**（想定ペルソナ／想定検索意図／タイトルとの整合性ポイント）

**見出し構造のルール**

- 使用するのはH2とH3のみ（H1＝確定タイトルは提案不要）
- 冒頭H2＝結論（記事の核となる答え）→中盤＝詳細・根拠（表・リストを優先）→終盤＝補足・周辺情報・内部リンク導線、という流れを基本とする
- **最低1つのH2は質問形にする**
- Step 1〜2で特定した「強い検索ニーズ」（例：出演者は誰か／何時から／見逃し配信の有無）には、必ず見出しレベルで直接応える構造にする
- 出演者・企画内容を扱う記事では、出演者一覧に直接応える見出しを最低1つ設ける
- タイムテーブル・出演時間等が未発表の場合は「現時点で分かっている範囲」「発表されている範囲」など、断定・誤解を避ける安全な表現を見出しに使う
- FAQはまとめ直前に必ずH2として設置し、内部はH3で3〜4問（番組内容と検索意図に基づく自然で実用的な質問にする）
- H3は必要な場合のみ使用し、使うなら1つのH2に対し2つ以上設ける
- 最後は必ず「H2 まとめ」で締め、まとめにH3は付けない

**文体ルール（見出し・メタディスクリプション共通）**
- 親しみやすく読者に寄り添うトーン。過剰なテンション・煽り表現は禁止
- 高校生でも理解できる一般的な日本語、日常会話になじみのある言葉を優先
- 「なお」「一方で」「本記事では」等の硬い接続語の多用を避ける
- スラング・極端な感情表現は禁止。語尾や言い回しが単調にならないようにする
- 整理済み素材（Step 1・資料①）に無い事実は断定しない

### Step 5〜6. 本文執筆・構造化データ・保存（1回のWriteで完結させる）

Step 3・4で確定したタイトル・構成の時点で、本文（Step 5）と構造化データ（Step 6）に必要な材料はすべて揃っている。**本文を保存してから構造化データを追記する、という2段階のEditに分けず、1回のWrite/Editで両方を含んだファイルを保存する。**

### Step 5. 本文執筆

**保存ルール**
- ファイル名: 単発テーマ記事は `draft_kw_{キーワードslug}.txt`。**毎年恒例の企画・コーナー記事（リライトの可能性がある場合）は年号を付けず `draft_{企画略称}.txt`** とし、番組特番draft群と同じく来年以降も上書き運用する（例：`draft_dancebattle.txt`）
- 保存先はこのフォルダ直下。CLAUDE.mdの「ドラフトファイルの命名規則」表に追記する
- 公開が完了したdraftファイルは、蓄積し続ける必要はない。記事本体はWordPress側が正であり、ローカルのdraft.txtは編集中の作業ファイルに過ぎない。ただし毎年恒例のコーナー記事は次回リライトのベースになるため残しておく

**5-A. リード文（記事冒頭）**

- 3〜5文・全角110〜150字前後。メインキーワードを1〜2文目に自然に含める。冒頭は読者の状況に寄り添う一文から始め、最後の一文で「この記事で分かること」を言い切る。感情表現は最大2回まで（例：「楽しみですね」）
- 未発表項目がある場合は「未発表」である旨を一度だけ明示する
- 冒頭のクイック情報ボックスや各H2直下の補足ボックス（`<aside>`/`<div>`のHTML装飾）は、固定テンプレートを機械的に流用するのではなく、その記事の内容・トーンに合わせて**臨機応変にデザインを生成する**（`draft_dancebattle.txt`で採用したパネル型デザインを標準的な出発点にしてよい）。以下は生成時に必ず守る技術ルール：
  - 装飾用の`<aside>`・`<div>`ブロックは `<!-- wp:html -->...<!-- /wp:html -->` で囲む（`wp:paragraph`では囲まない）
  - すべての`<aside>`・カード状`<div>`に `role="region" aria-label="..."` を付与する
  - スマホ表示を想定し、固定pxでの横幅指定は避け、`width:100%`・`box-sizing:border-box`・長い文字列には`word-break`などレスポンシブ対応を入れる
  - `<style>`タグを含むブロック（表のレスポンシブ切り替え等）も`wp:html`で囲む
  - ショートコード（`[nopc][title][/nopc]`等）とタイトル・メタディスクリプション行は独立した`wp:paragraph`ブロックに置く（[[feedback_cdtv_shortcode_blocks]]）

**5-B. 本文**

- H2は1つずつではなく通しで執筆してよい（途中で「次の章に進めてよいですか」等の確認は不要）
- 文字数目安：H2単体は最大400字／H2+H3構成のリード部分は90〜110字／H3は220〜280字／最終H2（まとめ）は最大400字（目安であり厳密でなくてよい）
- 表・箇条書き・引用ブロック等の視覚要素は必須にしない。SEO_guide.txtの原則（3つ以上の事象は表/リスト化）を満たす箇所があれば使う程度でよい
- ですます調で統一。高校生でも理解できる一般的な日本語。教科書的な説明文・ニュース原稿調を避ける
- 感情表現は各見出しにつき1〜2か所まで。過度に感情を煽るコピー調・主観的な称賛の多用・根拠のない期待を持たせる表現・ポエム調の比喩は禁止
- 未確定情報は断定せず「〜と考えられます」「〜という印象です」等で締める。噂・視聴者の声に触れる場合は「公式発表ではありません」等、距離を取った表現にする。匿名掲示板・出典不明のSNS情報は使用しない
- タイトル・メタディスクリプション行は `<!-- wp:paragraph -->` ブロックで囲む（[[feedback_shira_title_meta_block_wrap]]）
- 全リライト共通の禁止ワード（「彩る」「幕を開ける」等）を避ける（[[feedback_shira_rewrite_banned_words]]）
- 締め文に主観形容詞を使わない

### Step 6. 構造化データ生成（JSON-LD）

本文完成後、「まとめ」セクションより下・ファイル末尾に配置する。`draft_dancebattle.txt`で確立した形式を標準とする。

**固定要件**
- `<!-- wp:html -->` で囲み、内部に `<!-- MANUAL_JSONLD -->` → `<script type="application/ld+json">` → JSON（`@graph`形式）→ `</script>` → `<!-- /wp:html -->`
- `@graph`で全スキーマを統合し、日付はISO8601（JST +09:00）
- `@id`は `https://shira-treat.com/{スラッグ}/#post` のようにセクションIDで統一
- `keywords`は配列。画像は`width:1200, height:675`で統一し`thumbnailUrl`も同一URL

**固定データ（毎回そのまま入れる）**
- `publisher`: Organization / name: "Shira Notes" / url: `https://shira-treat.com/` / sameAs: `["https://x.com/ShiraNotes_"]` / logo: ImageObject `https://shira-treat.com/wp-content/uploads/2025/08/Shira-Notes-1.webp`（512×512）
- `author`: Person / name: "Shira Notes" / url: `https://shira-treat.com/operator-information/`
- `inLanguage`: "ja"

**スキーマ構成（出力順固定）**
1. `BlogPosting`（必須）：headline/description（メタディスクリプションと一字一句一致させる）/keywords/datePublished・dateCreated・dateModified/image/author/publisher/articleSection/about（**必ず`Thing`型。Event/Product/Personは禁止**）/mentions（任意）/hasPart（配下スキーマの`@id`一覧）
2. `BroadcastEvent`（放送企画の記事の場合）：startDate・endDate・location必須。eventStatus・eventAttendanceMode・isLiveBroadcastも設定。物理会場が未公表の場合は`location`を`VirtualLocation`（公式サイトURL参照）にする。performerは本文に実際に記載されている出演者・グループのみを反映（推測禁止）。配信ありなら`potentialAction`に`WatchAction`を設定
3. `ItemList`（出演者・企画一覧がある場合）
4. `FAQPage`：本文FAQの内容のみを反映する（推測禁止。不明点は本文同様「公式発表なし」の表現に合わせる）
5. `BreadcrumbList`：実在するURLのみ使用。カテゴリURLが未確定の場合は「ホーム→記事→本記事」の3階層（[[project_shira_utadeaetara]]と同じ`draft_ongakunohi.txt`の簡易パターン）を使う

**品質チェック**：`@id`の重複がないか、`about`が`Thing`型のみか、メタディスクリプションと`description`が一字一句一致しているか（`qa_draft.py`のERRORチェック対象）を確認する。

### Step 7. 検品（確認を挟まず自動実行する）

本文＋構造化データを保存したら、**「検品しますか？」とユーザーに確認を取らず、そのまま`python tools/qa_draft.py draft_kw_{キーワードslug}.txt`を自動実行する。**

- ERROR/WARNが出た場合は、その場で修正して再実行する（ユーザーには修正前の中間状態を見せない）
- ERROR 0件になった時点で初めてユーザーに完了報告する。報告には「何を直したか」を簡潔に添える（例：attributionsrc属性・メタディスクリプション行フォーマット等）
- 番組タイムテーブル特有のチェック（アーカイブパネル等）は該当しないため無視してよいが、スマートクォート・禁止ワード・WPブロック開閉・JSON-LDパース・メタディスクリプション⇔JSON-LD description一致は共通適用される

## 補足：番組タイムテーブル速報との違い

| 項目 | タイムテーブル速報draft | キーワード起点記事 |
|---|---|---|
| 起点 | 番組の放送予定・資料① | 狙いキーワード |
| 先頭H2 | 必ず「最新タイムテーブル」 | 記事の結論（キーワードにより可変） |
| 更新運用 | 毎回上書きリライト | 基本は単発（情報更新時のみ加筆） |
| ファイル命名 | `draft_{番組略称}.txt` | `draft_kw_{キーワードslug}.txt` |
