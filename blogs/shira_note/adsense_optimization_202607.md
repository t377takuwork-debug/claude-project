## 改善ポイント（優先度順）

### 🔵 優先度：格下げ（要検証だが緊急ではない）：全画面広告「追加のトリガー」の見直し

**2026/7/25追記**：3期分のGA4データ（新規・リピーターユーザー数）で検証した結果、リピーター比率は14.5%→14.5%→15.9%とトリガー導入前後で悪化しておらず、むしろ横ばい〜微増。絶対数の変動は年末特番シーズン→閑散期→通常編成という季節性で説明がつく動きだった。この指標からは「追加のトリガーが再訪・エンゲージメントを悪化させている」証拠は出ていない。優先度は下げるが、低リスク・可逆的な変更のため試す価値自体は残る。

- 現在オン。全画面広告の頻度は「2分」に設定（デフォルトは10分、範囲は1分〜1時間で収益最大化狙いの設定）。
- しかし平均エンゲージメント時間は38秒。2分の時間ベース発火はほとんどのセッションで作動しないため、**現在出ている全画面広告の大半は時間経過ではなくアクション起点のトリガーで発火している可能性が高い**。
- 「追加のトリガー」に含まれる操作：戻るボタンで前ページに戻る／記事末尾まで読んで上にスクロールし始める／タブ・ウィンドウの再表示／ブラウザのナビゲーションバー操作／30秒以上操作なしの後の再操作。
- このうち「戻るボタン」と「上スクロール開始」は、タイムテーブル記事で最も典型的な離脱・見返しパターンと重なる（検索から来て確認後に戻る／下まで読んで上の時間帯を見返す）。**利用シーンの最後に必ず広告が割り込む設計**になっており、放送中に何度もチェックし直すこの手のサイトでは再訪意欲を最も削りやすい。
- この機能は2026年2月9日追加、対応しないと3月9日に自動有効化される仕様。意図的にオンにした記憶がなければ、自動適用されたものである可能性が高い。

**アクション**：
1. 「全画面広告の追加のトリガーを許可する」をOFFにする（頻度2分はそのまま維持してもよい）。
2. GA4で2026年2月以前（トリガー導入前）と4月以降（導入後）の直帰率・平均エンゲージメント時間・再訪問率を比較する（自然実験として効果を検証できる）。
3. OFFにした後、2週間程度で収益・エンゲージメント両方の変化を再測定する。

### 🟠 優先度高：モバイル広告のサイズを自動で最適化（現在OFF）

- 収益の96%を稼ぐハイエンドモバイル向けの最適化が未実施。
- モバイル向けページとAMPで、既存の広告ユニットサイズをデバイスの向き・文脈に応じて自動拡大する仕組み。新規タグ設置不要。
- 制約：iframe広告ユニット、サイズ固定コンテナ、ユーザーの初回ページビューには非対応。

**アクション**：「広告」→「全般設定」→「モバイル広告のサイズを自動で最適化する」をON。

### 🟡 試す価値あり：ページ内広告（バナー／Multiplex）の新規有効化

- 自動広告のうち「ページ内フォーマット」（バナー広告・Multiplex広告・関連する検索）は3種とも完全にOFF。収益源が割り込み型に偏っている今、非侵入型の収益源を追加できる余地がある。
- 「既存の広告を最適化する」はOFFのまま維持し、手動配置（タイトル下48%・目次24%という高パフォーマンス箇所）にGoogleが介入しないようにする。
- 「ページに表示する広告の最大数」「最小間隔」は最小値から開始し、様子を見ながら段階的に調整する。

### 🟡 試す価値あり：自動最適化（現在OFF）

- サイト単位でON/OFF可能。ONにするとトラフィックの一部（デフォルト50%、現状スライダーは25%に設定済み）で自動広告フォーマットの改善テストが実施される。
- テスト終了後に通知が届き、設定を変更していなければ元に戻せるため下振れリスクは限定的。

**アクション**：ONにして経過観察。

**2026/7/25追記：自動最適化の詳細設定について**

自動最適化をONにした後、以下2点の調整が必要と判断。

1. **「テスト結果で優位なパターンを自動適用する」はOFFにする**（現状ON）。Googleの「有利」判定がこのサイトの文脈（速報系・短時間セッション）でも本当に有利とは限らないため、テスト結果は通知を見て内容を確認してから手動で適用する。
2. **「ブロック中のテスト」に広告インテントを追加でブロックする**（現状未ブロック）。未ブロックのままだと、`google-anno-skip`でセットリスト表を除外する準備ができる前に、Googleが自動でテスト→自動適用してしまうリスクがある。
3. 既存のブロック項目（アンカー広告・バナー広告・バナー広告の調整・全画面広告・全画面広告の表示頻度）はそのまま維持。これらは既にこちらでチューニング済みの設定であり、変更する場合もGA4と突き合わせて自分たちで検証すべき。
4. ブロックされていないサイドレール広告・ワイドスクリーンアンカー広告・ワイドスクリーン全画面広告は許可のままでよい（デスクトップ関連で収益影響3.3%と小さいため）。
5. テストのトラフィック割合（25%）は変更不要。上記の通り高リスク項目は既にブロック済みのため、残るテスト対象は影響の小さいフォーマットのみ。

### 🟠 優先度高（確定）：手動Multiplexユニットの配置変更

期間2025年11月1日〜2026年7月21日の実測値で原因が確定した。

| 指標 | 値 |
|---|---|
| 表示回数 | 649,774 |
| クリック数 | 1,007 |
| CTR | 0.155% |
| RPM | ¥15.36 |
| CPC | ¥9.91 |
| アクティブビュー視認可能率 | 12.61% |
| 推定収益額 | ¥9,983 |

- **原因は広告の質ではなく視認性**。CPC(¥9.91)は他の手動ユニット平均(全体で約¥2.67)より高く、クリックされれば単価は悪くない。問題はCTRが0.155%と極端に低いこと。
- Google公式の目安では視認可能率50〜60%が適正な広告配置、ネットワーク平均は約50.2%（[アクティブ ビュー視認可能率 - Google AdSense ヘルプ](https://support.google.com/adsense/answer/3432842?hl=ja)）。12.61%はその目安の4分の1以下。
- 配置は「記事末」。平均エンゲージメント時間38秒と重ね合わせると、ほとんどの読者が記事末尾に到達する前に離脱しており、そもそも画面に表示される機会自体が少ないと考えられる（タイトル下・目次が手動収益の72%を稼いでいるのも、上部＝高視認性という同じ理由で説明がつく）。

**アクション**：Multiplexを記事末から、セットリスト/タイムテーブル表の直後など読者が実際に読み進めている中盤に移動する。ユニット自体の差し替えより配置変更を優先する（CPCは既に良好なため）。

**2026/7/25追記：配置場所の注意点**

未設定のレスポンシブMultiplexは、モバイルで「4行1列」など縦積みレイアウトで表示されるのが標準（[レスポンシブ対応のMultiplex広告ユニットをカスタマイズする方法](https://support.google.com/adsense/answer/7533385?hl=ja)）。文章の途中（段落と段落の間）に差し込むと縦に長く読書の流れを断ち切るため、**「本編（セットリスト）が終わった直後、関連情報セクションへの切れ目」に置く**のが正しい。この位置なら「本編は読まれている＝視認される確率が高い」かつ「切れ目なので縦長でも違和感がない」の両立ができる。行数・列数を`data-matched-content-rows-num`等で個別調整する方法はGoogle公式に「サポート終了」と明記されており、恒久対策としては使わない。

**2026/7/25追記：cdtv-timetableページのスマホ実機画面で検証**

`https://shira-treat.com/cdtv-timetable/` のスマホ表示（印刷PDF）を分割して確認した結果：

- ページ全体の長さは約24画面分（iPhone基準）。平均エンゲージメント時間38秒では大半のユーザーが最後まで到達しない。
- 核となる「CDTVライブ！ライブ！実際の出演順【放送後更新】」（本編セットリスト）は、ページ全体の**上から約40〜45%の位置**に出現する。
- 現在のMultiplexは、SNSシェア→関連記事→アーカイブナビ→検索ボックス→著者プロフィール→最近の投稿、という一連の要素の後、**ページ全体の70〜85%の深さ**に配置されている。視認可能率12.61%はこの位置の深さで説明がつく。
- **移動先案**：「実際の出演順」テーブル直後（見逃し配信セクションの前後）。本編を読み終えた読者がまだ残っている可能性が高く、かつ本編とアーカイブ情報の切れ目に当たるため縦長でも違和感がない。現在地より体感8〜10画面分手前への移動になる。

**2026/7/25追記：実際の配置をユーザーに確認・訂正**

画面確認後、ユーザーから実際の配置の正確な言語化が得られた。想定より早い段階に配置されていたため、以下の通り訂正する。

現状の正確な配置（ページ上から順）：

| 順序 | 位置 | ユニット |
|---|---|---|
| 1 | アイキャッチ画像の直後（冒頭ボックスの上） | タイトル下 |
| 2 | 冒頭ボックス（ステータス表示）の直下 | 目次 |
| 3 | 「予想タイムテーブル」ボックスの直後 | original（1回目） |
| 4 | 「後半ほど注目アーティストが配置される傾向です」の直下 | original（2回目） |
| 5 | Amazon Musicリンクの下（見逃し配信セクション末尾） | Multiplex |

**本当の問題**：Multiplexは記事の最後ではなく、5つの広告チェックポイントのうち「最後（5番目）」に位置している。originalが2回（3・4番目）を占有し、Multiplexは一連の広告枠の中で最も離脱後に残る読者が少ないポジションに置かれている。平均エンゲージメント38秒という短さでは、チェックポイントが進むごとに残存読者が急減するため、絶対位置よりも「何番目のチェックポイントか」が視認可能率を左右している可能性が高い。

**最終提案（広告枠を増やさず順番を入れ替え）**：originalの2回目（4番目の位置＝「実際の出演順」が明かされる直前）をMultiplexに差し替える。

| 順序 | 位置 | ユニット | 変更 |
|---|---|---|---|
| 1 | アイキャッチ画像の直後 | タイトル下 | 変更なし |
| 2 | 冒頭ボックスの直下 | 目次 | 変更なし |
| 3 | 「予想タイムテーブル」ボックスの直後 | original | 変更なし（1回のみに整理） |
| 4 | 「後半ほど注目アーティストが配置される傾向です」の直下 | Multiplex | originalから差し替え |
| 5（削除） | Amazon Musicリンクの下 | なし | 広告を外す |

利点：①本編の出演順が明かされる直前という関心の高いタイミングにMultiplexが繰り上がる、②originalの重複表示（カニバリ懸念）も同時に解消、③広告総数は増えないためページがこれ以上長くならない。

**その他、画面確認で気づいた点（参考・優先度低）**：
- 「累計140万件突破【フラット35】」の住宅ローンバナー広告が、目次直前と予想タイムテーブル直後の近接した2箇所にほぼ同じ内容で重複表示されている。カニバリの可能性があるため間隔を空けることを検討。
- アイキャッチ画像直後にチャット風ネイティブ広告（「HYZE」）が設置されており、本文と誤認されやすい位置。意図した配置か確認推奨。

- 本文中の既存テキストをリンク化し、クリック時にダイアログで広告＋Gemini生成コンテンツを表示する機能。

**アクション**：導入する場合は、セットリスト表など核となる情報部分に `google-anno-skip` クラスを付けて対象外にし、本文の解説文のみで限定テストする。
