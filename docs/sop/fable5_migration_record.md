## 現在の運用フロー（概要）

ユーザーの投げ方は従来と完全同一。司令塔が資料整形・承認取得→エージェントへ委譲→機械QA合格の報告→司令塔がgit diff＋QA再実行＋主張の独立裏取りで受入→コミット→失敗パターンをrules/メモリへ昇格。6ステップの詳細と**委譲プロンプト定型**は `sop_agent_delegation.md` を参照。

## 注意点・既知の制限（2026-07-19時点）

1. **ネイティブagentタイプ未確認**：shira-rewriter／post-writerは作成セッション内では一覧に載らず、パイロット2回ともフォールバック方式（general-purpose＋sonnet＋定義貼り込み）で実施。新セッションでの認識確認が未実施
2. **frontmatterの`effort: medium`キーは実効性未検証**（無効でも無害）
3. **書き込みスコープ制限は定義レベル**：ツール機構でのパス制限は不可のため、定義の明示＋受入時git diff検分で担保する
4. **本環境のBashツールは出力不達**：委譲文に「PowerShellを使え」を毎回含める（SOP記載）
5. **サブエージェントはSkill起動・ユーザー往復が不可**：LLMレビュー（/quality-guardrail・/post-review）は司令塔工程、ユーザー承認は「停止→司令塔経由」方式で代替
6. **エンジン移行済みはutaconのみ**：残り10番組は従来フル手順のまま（移行期ルールで共存中）
