# 運用マスターシート — Fable 5体制の completeガイド

作成: 2026-07-19。**このシート1枚で日常運用のすべてが完結する**ことを目的とした運用者向け完全版。
体系上の唯一の正は `sop_agent_delegation.md`（体制）と各agent定義・エンジン・ルールファイル。本シートと食い違ったらそちらが正であり、更新時は両方を同期する。

## 1. 体制全体像

### 役割分担（1行で）

**機械QAで合否判定できる作業はSonnetのエージェントに委譲。判定できない作業（戦略・設計・環境変更・例外処理）はFable 5が直接やる。**

| 立場 | 誰 | やること |
|---|---|---|
| オーナー | あなた | タスクと資料を渡す・設計テーブルや整形案を承認する・最終確認 |
| 司令塔 | Fable 5（メインセッション） | 資料の整形と事前チェック→委譲→受入判定→コミット→失敗パターンの昇格 |
| 実行部隊 | Sonnet 5（エージェント） | 手順書どおりの生成・編集→機械QA合格まで。判断に迷ったら停止して司令塔に返す |

### 資産の階層（どこに何があるか）

```
README.md                     ← 全体索引（迷ったらここから）
CLAUDE.md                     ← 行動規範・タスク別エントリポイント
docs/sop/
  ├ sop_agent_delegation.md   ← 委譲体制の正（フロー・注意点・ロードマップ）
  ├ fable5_migration_record.md ← 移行の経緯記録（読み物）
  └ 運用マスターシート.md      ← 本シート
.claude/agents/               ← 実行部隊の定義（shira-rewriter / post-writer）
.claude/commands/             ← 汎用スキル22本（/note-article 等）
blogs/shira_note/
  ├ rewrite_common_rules.md   ← リライト共通ルール（唯一の正）
  ├ rewrite_engine.md         ← 速報型リライトの共通手順骨格
  ├ .claude/commands/         ← 番組別ファイル15本（ENGINE準拠 or フル手順）
  └ tools/qa_draft.py         ← 機械検品（リライトの合否判定機）
brands/tools/qa_post.py・qa_article.py ← SNS投稿・Note記事の合否判定機
メモリ（~/.claude/projects/...) ← セッション横断の記憶。週1精製
```

## 2. 日常運用の完全フロー

### A. shira_note 番組リライト（最頻・委譲する）

1. **あなた→司令塔**：番組名＋資料①（放送日・出演者・曲・見どころ）＋資料②（前回の確定データ）＋資料③（視聴メモ・任意）＋アイキャッチを渡す
2. **司令塔の事前チェック**（自動でやる）：資料②をテンプレ表に整形／時間帯と放送枠の整合確認（ズレ・欠落があれば選択肢つきで確認が来る→**承認するだけ**）／通常回か特番か／前回が特別回なら「戻し」を委譲文に明記
3. **委譲**：司令塔がshira-rewriterへ投げる
4. **実行部隊**：共通ルール→エンジン→番組別ファイルの順に読み、Editで差分更新→`qa_draft.py` ERROR 0件・新規WARN 0件・exit=0まで
5. **司令塔の受入**：git diff（変更が指示範囲だけか）＋QA再実行＋事実主張の裏取り（例: YouTube動画IDはRSSを自分で再取得して確認）→合格ならコミット
6. **昇格**：報告に「初回不合格ログ」や「保留・提案」があれば、ルール化やメモリ記録の要否を司令塔が判定（実バグ発見の実績あり）

**あなたが打つプロンプト例**：
> shira-rewriterに委譲：うたコン 7/28回のリライト。
> 資料①：〔放送日・出演者・見どころ〕 資料②：〔前回データ〕 アイキャッチ：既存URL維持。
> 受入は git diff＋qa_draft.py exit=0 で。

### B. SNS投稿バッチ（MBTICODE / s4lv・委譲する）

1. **あなた→司令塔**：「post-writerでMBTICODE明日分（X・Threads各3本）」のように依頼
2. **Phase A**：エージェントが事前設計テーブル（柱・FW・狙う反響）を提示して**止まる**→あなたが承認
3. **Phase B**：本文生成→`qa_post.py` ERROR 0件・exit=0→`posts_*.txt`へ保存→報告
4. **司令塔の受入**：QA再実行＋git diff（posts_*.txt以外を触っていないか）＋**/quality-guardrail・/post-review（LLMレビューは司令塔の仕事。エージェントは実施できない）**
5. 合格→コミット。あなたはpostsファイルからコピペ投稿

### C. 司令塔が直接やる作業（委譲しない）

| 作業 | 起動方法 |
|---|---|
| 戦略・方針の相談 | `/notekaigi`（構造的な判断は必ずここを通す） |
| Note記事生成 | `/note-article mbticode 〔テーマ〕`（機械QA=qa_article.py＋/quality-guardrail） |
| 新規番組記事の立ち上げ | `/shira-new-article`（コマンド新設・CLAUDE.md登録まで1セッション） |
| ASP記事設計 | `/asp-kaigi`→`/asp-theme`→`/asp-outline` |
| 環境変更（フック・settings・スケジューラ） | 依頼文で直接指示（機械QAがないため委譲対象外） |

## 3. Agent・Skill・エンジン一覧

### エージェント（実行部隊）

| 名前 | できること | 使うタイミング | 受入基準 |
|---|---|---|---|
| **shira-rewriter** | 番組リライトのEdit差分更新→QA合格まで。Write（全体上書き）・git・rules編集は禁止設計 | 番組リライトを頼むとき毎回 | qa_draft.py ERROR 0・新規WARN 0・exit=0 |
| **post-writer** | 投稿バッチの設計→生成→QA→posts_*.txt保存。書き込み先は4ファイル限定 | 投稿バッチを頼むとき毎回（※初回パイロット未実施） | qa_post.py ERROR 0・exit=0＋司令塔のLLMレビュー |

※エージェント名が認識されない場合：司令塔が自動で代替起動する（`.claude/agents/`の定義を汎用エージェント＋Sonnet指定に貼り込む方式。パイロット2回の実績あり）

### rewrite_engine.md（エージェントではなく「共通手順書」）

速報型リライトの手順骨格1〜14（退避→タイトル・メタ→aside→本文H2群→アーカイブ→過去一覧→FAQ→まとめ→JSON-LD→QA）。**ENGINE準拠番組（現在うたコンのみ）**はこれ＋軽量な番組パラメータで動き、未移行番組（Mステ等10本）は従来のフル手順ファイルが正。共通ルールの変更が1ファイルの修正で済むのが利点。

### 主要スキル（よく使う順）

| スキル | 用途 |
|---|---|
| `/shira-qa` | draft検品の起動と結果解釈（リライト後必須） |
| `/post-review` | SNS投稿の冷徹壁打ち（投稿確定前に必須） |
| `/quality-guardrail` | AIっぽい薄い表現の検出・添削（Note記事・投稿） |
| `/mbticode-post`・`/s4lv-post`・`/reply` | 投稿・リプライ生成（post-writer委譲に移行中） |
| `/note-article`・`/junk-theme`・`/junk-article` | Note記事の一貫生成 |
| `/cf-article`・`/vtuber-article` | ブログ記事生成（画像自動配置つき） |
| `/notekaigi`・`/blog-kaigi`・`/monetize-kaigi`・`/asp-*` | 会議・戦略系 |
| `/buzz-analysis`・`/kpi-weekly` | 分析・記録 |
| `/consolidate-memory` | 週1メモリ精製 |
| `/blog-launch`・`/shira-new-article` | 新ブログ・新番組の立ち上げ |

## 4. あなた（司令塔含む）が必ずやること・タイミング

| タイミング | 作業 | 補足 |
|---|---|---|
| 毎朝 | 「前回のネタ結果を見せて」 | スケジューラが7:00/7:10に自動収集済み。再実行不要 |
| タスク発生時 | 資料を揃えて依頼（捏造禁止のため資料が正の源） | 不足があれば司令塔から質問が来る |
| 承認要求が来たら | 整形案・設計テーブルをOK/修正 | 放送枠ズレ等は選択肢つきで来る |
| 委譲の受入後 | コミットは司令塔がやる（勝手にはやらない運用ならその旨指示） | 実行部隊はgit禁止 |
| 週1 | `/consolidate-memory`（メモリ精製）・`/kpi-weekly`（KPI記録） | 精製は削除前バックアップ必須の安全設計 |
| ルール変更時 | repo内ファイルだけを編集（メモリ・旧マスターは触らない） | 本数などの記載はREADMEと実態を同期 |
| 失敗発生時 | 「初回不合格ログ」「保留・提案」の昇格判定を司令塔に求める | ルール化された失敗は再発しない |

## 5. 注意点・禁止事項・よくある失敗パターン

### 絶対禁止

- **捏造**：実体験・実績・出演者・時刻は `personal_data.md`／`reference/`／渡した資料にあるものだけ。**無いものは「不明」と書くのが正解**（時刻の創作は過去に手戻り実績あり）
- **`--update-baseline`**：QAの基準変更はあなたの承認事項。エージェント・司令塔とも勝手に実行しない
- **`git reset --hard`・`rm -rf` 等**：settings.jsonのdeny＋Guardrailフックでブロック済み（意図的に外さない）

### よくある失敗と回避策（実績ベース）

| 失敗パターン | 回避策（実装済みの仕組み） |
|---|---|
| 退避データ（前々回）と資料②（前回）の取り違え→過去一覧が重複 | エンジン手順1'に明文化＋完了後の日付セルフチェック。新規立ち上げ時は「過去一覧の先頭は前々回から」 |
| 特別回→通常回戻しで特別回名が残存 | 委譲文に「戻し」を明記し、Grepで全箇所洗い出してから除去 |
| 資料②が自由形式のまま処理されて推測混入 | 司令塔が必ずテンプレ整形→あなたの承認を取ってから委譲 |
| 「できたはず」報告 | 受入は必ず実出力（QA結果・exit値・git diff）で判定。出力なしの合格主張は無効 |
| 同じ修正の無限ループ | 2回失敗したら3回目をやらず停止・報告（agent定義に組み込み済み） |
| 自動収集ファイル（tools/output/）が作業コミットに混入 | 受入コミット時に対象ファイルだけをgit addする |

### 環境の既知制限

- **Bashツールは出力が返らない**→常にPowerShell（委譲文にも毎回明記）
- **.ps1に日本語を書くと文字化けで壊れる**→スクリプトはASCIIのみ
- **フック・スケジューラのpythonは絶対パス必須**（`C:\Users\PC_User\AppData\Local\Python\bin\python.exe`）
- **エージェントの書き込み制限は「定義＋git diff検分」で担保**（ツール機構ではパス制限できない）
- **新設エージェントは作成したセッションでは名前起動できない**→新セッションで確認（フォールバックあり）

## 7. トラブル時の対応フロー

| 症状 | 対応 |
|---|---|
| QAが2回直しても通らない | エージェントは自動停止して報告してくる→「現状・試したこと・仮説」を見て司令塔に方針転換を指示（3回目の変種はやらせない） |
| スラッシュコマンドが動かない | `.claude/commands/`はgit管理下。`git log -- .claude/commands/<ファイル>` → `git show <コミット>:<パス>` で過去版復元 |
| draftを壊した気がする | `git diff drafts/draft_XXX.txt` で差分確認→問題なら `git restore <ファイル>`（reset --hardは使わない） |
| エージェントの暴走が疑わしい | まずgit diffで実害確認（実行部隊はgit禁止なので未コミット差分に必ず残る）→不要ならrestore→定義の禁止事項を強化 |
| 朝の自動収集が動いていない | `Get-ScheduledTaskInfo -TaskName "ShiraNotes_CollectNews"` でLastTaskResult確認（0が正常）→再登録手順は `blogs/shira_note/CLAUDE.md` に記載 |
| フックが効いていない | `.claude/settings.json` のexecフォーム（args指定）と絶対パスpythonを確認（シェル文字列型は600秒ハングする既知問題） |
| メモリと実態が食い違う | repo側が常に正。メモリは`/consolidate-memory`で修正・削除 |
| どこを見ればいいか分からない | README.md → 該当CLAUDE.md → 本シート の順。それでも不明なら司令塔に「〔タスク名〕の正しい入口は？」と聞く |
