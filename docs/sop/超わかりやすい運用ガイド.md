# 超わかりやすい運用ガイド

このガイドは「毎日どう動けばいいか」だけを書いたものです。難しい言葉は使いません。
もっと詳しい情報がほしいときは `運用マスターシート.md` を見てください。

---

## 1. 体制ってなに？（かんたんに）

あなたの下に、2種類のAIがいると思ってください。

| 名前 | たとえるなら | やること |
|---|---|---|
| **司令塔（Fable 5）** | あなたの秘書 | あなたの依頼を受け取り、資料を整えて、実際の作業を部下に頼む。できあがりをチェックしてから提出する |
| **実行部隊（Sonnet 5）** | 秘書の部下 | 決まった手順どおりに実際の作業（記事の書き換え・投稿文の作成）をする。機械のチェックに合格するまでやる |

**あなたがやることは1つだけ**：司令塔（＝今話しているこのAI）に、必要な情報を渡して頼むこと。あとは司令塔が部下に振ってくれます。

## 2. 毎日・毎週やること（チェックリスト）

### 毎朝

- [ ] 「前回のネタ結果を見せて」と言う
  → 朝7時ごろに自動で番組情報が集まっているので、それを見るだけ。自分で調べなくてOK

### 何かを頼むとき（毎回）

- [ ] 必要な資料をそろえてから頼む（資料が足りないと聞き返されます）
- [ ] 司令塔から「これでいいですか？」と聞かれたら、確認してOKを出す
- [ ] できあがったら「ちゃんとチェック（QA）通った？」の一言だけ確認する

### 週に1回

- [ ] 「メモリを整理して」と言う（`/consolidate-memory`）→ AIの記憶を整頓してくれる
- [ ] KPI（数字の実績）があれば「KPI記録して」と渡す（`/kpi-weekly`）

## 3. 主な道具（ツール）一覧

### 3-1. shira-rewriter（番組リライトの担当）

- **何ができる？** 音楽番組のタイムテーブル記事を、最新の放送内容に書き換えてくれる
- **いつ使う？** 番組の出演者・曲目が分かったとき（放送前・放送後どちらも）
- **どうやって使う？**
  ```
  shira-rewriterに委譲：うたコン 7/28回のリライト。
  資料①：放送日・出演者・見どころ
  資料②：前回の確定データ
  アイキャッチ：既存URL維持
  ```
  → これをそのまま貼るだけでOK。あとは司令塔と部下がやってくれます

### 3-2. post-writer（SNS投稿の担当）

- **何ができる？** X・Threadsの投稿文をまとめて作ってくれる
- **いつ使う？** 毎日の投稿バッチを作りたいとき
- **どうやって使う？**
  ```
  post-writerで、MBTICODEの明日分（X・Threads各3本）を作って。
  ```
  → 途中で「こんな内容で作りますが良いですか？」と一覧が出てくるので、OKを出すと完成する

### 3-3. よく使うそのほかの機能

| 名前 | 何ができる？ | 使うとき |
|---|---|---|
| `/notekaigi` | 方針や戦略をみんなで話し合って決めてくれる | 「これからどうしよう」と迷ったとき |
| `/note-article` | Note記事を最初から最後まで作ってくれる | Note記事を書きたいとき |
| `/post-review` | 投稿文を厳しくチェックしてくれる | 投稿を出す直前の最終確認 |
| `/quality-guardrail` | 「AIっぽい薄い文章」を見つけて直してくれる | 記事・投稿を作ったあと |
| `/shira-qa` | リライトした記事にミスがないかチェックする | リライトのあと（自動でもやってくれる） |

## 4. こんなときどうする？（タスク別ガイド）

### 番組のリライトを頼みたいとき

1. 放送日・出演者・曲目・見どころをメモしておく（分かる範囲でOK）
2. 前回の放送データ（時間帯・出演者・曲）があれば一緒に渡す
3. 司令塔に「〇〇のリライトお願い」と伝える
4. 資料が変な形（時間がバラバラ等）だと、司令塔が「こう直していいですか？」と聞いてくるので確認する
5. 完成の連絡が来たら終わり（QAチェック済みのものが出てきます）

### 投稿を作りたいとき

1. 「MBTICODE（またはs4lv）の投稿を〇本作って」と伝える
2. 案が出てくるので内容を見てOKを出す
3. 本文ができあがったら、機械チェック→念のため最終チェック（/post-review）を通してから確定
4. 保存されたファイルからコピペして投稿する

### 記事を書きたいとき

1. 「Note記事作って、テーマは〇〇」と伝える（`/note-article`）
2. テーマ・タイトル案が出るので選ぶ
3. 本文ができたら保存場所を確認して完了

### 方針に迷ったとき

1. 「〇〇について会議して」と伝える（`/notekaigi`）
2. 複数の視点から検討した結果が返ってくる

## 5. 気をつけること・よくあるミス

### ぜったいにやってはいけないこと

- ❌ **ないものを「あることにして」書かせる**（実績・体験・出演時間など）
  → 分からないものは「不明」でOK。無理に埋めない
- ❌ **チェック（QA）を飛ばして「たぶん大丈夫」で確定する**
  → 必ずチェック結果（合格・件数）を確認してから完了とする

### よくあるミスと直し方

| こんなミスが起きがち | どうすればいい？ |
|---|---|
| 前回のデータと前々回のデータを混同してしまう | 資料を渡すときは「これは何月何日のデータです」と日付を明記する |
| 特別編成の回のあと、通常の回に戻すときに前の特別な文言が残る | 「前回は特番でした、今回は通常回です」と伝える |
| 資料がバラバラの形式で伝わる | 表形式（時間・アーティスト・曲）で渡すと一番スムーズ |
| 「できたと思います」で終わってしまう | 「チェック結果は？」と一言聞く癖をつける |
| 同じ直しを何度もやり直している | 2回直しても駄目なら、いったん止めて「どうしましょう？」と相談してもらう（自動でそうなる仕組みです） |

### もし何かおかしいと思ったら

- 「今の変更、大丈夫か確認して」と言えば、司令塔が変更内容を見せてくれます
- 「元に戻して」と言えば、保存前の状態に戻せます（勝手に消えることはありません）
