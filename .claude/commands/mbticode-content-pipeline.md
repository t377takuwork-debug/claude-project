# MBTICODE Content Pipeline

`mbticode/articles/published/` に保存された過去記事（原資）から、SNS投稿ネタを自動生成するルーチン。
`/mbticode-post` とは別物：あちらは週次バッチの本文生成、こちらは「記事という資産からネタを掘り出す」ネタ出し専用。

## 呼び出し方

```
/mbticode-content-pipeline
```

引数なし。実行するたびに `mbticode/posts/scheduled_posts.md` を上書き生成する。

---

## Step 0：上書き前の未消化チェック（必須）

`scheduled_posts.md` が既に存在する場合、生成を始める前に必ず内容を確認する。

1. 既存ファイル内の✅（来週使用候補）行それぞれについて、`posts_x.txt` / `posts_threads.txt` に本文化・保存済みかを確認する
2. **未消化（本文化されていない✅行がある）場合**：上書きしてよいか、または未消化分をどう扱うか（今回のプールに引き継ぐ／別ファイルに退避する／破棄する）をユーザーに確認してから進める。無断で上書きしない
3. 全て消化済み、またはファイルが存在しない場合はそのままStep 1へ進む

---

## 実行前に必ず読むファイル

1. `brands/mbticode/articles/published/*.md`（全記事・原資）
2. `brands/mbticode/articles/article_index.md`（管理No・URL対応）
3. `brands/mbticode/sns_post_cheatsheet.md`（型A/B/C/D・FW比率・禁止型・金言テスト・ブックマーク3動機・当たり前感解消ルール・トーン禁止事項）
4. `brands/mbticode/reference/`（MBTI・ラブタイプの特性根拠。ネタの理由づけに架空根拠を使わない）
5. `brands/mbticode/posts/posts_x.txt` / `posts_threads.txt`（直近投稿済み内容との重複回避）
6. `brands/mbticode/posts/cta_templates.md`（記事別CTAライブラリ。有料記事への誘導が狙えるネタか判断する際に参照する）

---

## Step 1：記事から「ネタの種」を抽出する

各記事本文を読み、SNS投稿1本の核になりうる「1行に圧縮できる主張・区別・逆転」を探す。

**抽出基準**
- 記事内の `>` blockquote・太字フレーズを優先候補にする（すでに核心として書かれている箇所のため）
- 記事1本につき2〜4個の種を抽出する（1記事1ネタに限定しない）
- 「AとBは別の話」型の区別、または「〇〇じゃなくて△△」の逆転構造を優先する
- 記事の有料/無料は種の採用可否に関係ない。ただし**有料記事(②③)の核心部分（第4章のテンプレ実例など）をそのまま引用しない**。構造・視点だけを借りて新しい言葉で書き直す

**除外基準（sns_post_cheatsheet.md 採用しない型と同一）**
- 攻撃的ラベリング・議論誘発型になる種は除外
- すでに `posts_x.txt` / `posts_threads.txt` に近い表現がある種は除外

---

## Step 2：型・FWを割り当てる

抽出した種それぞれに、`sns_post_cheatsheet.md` の型を1つ割り当てる。

| 割り当て項目 | 参照ルール |
|---|---|
| 型（A/B/C/D/α/二つの温度） | 種の構造に合う型を選ぶ。逆転構造が強い種は型α優先 |
| FW（MBTI／ラブタイプ／なし） | 記事内で言及されているタイプに合わせる。DSKBは月2本上限に注意 |
| 発話モード | 内声モード優先（型α・型B独り言型・二つの温度型）。解説モードは週3本上限ルールを踏まえて割り当てすぎない |

---

## Step 3：10件のアイデアプールを生成する

以下の形式で `mbticode/posts/scheduled_posts.md` に出力する。**本文を完成させず、投稿の核（金言フレーズ）と設計メモまでを出す**（完成本文化は `/mbticode-post` 実行時に行う）。

```markdown
# MBTICODE SNS投稿ネタ・アイデアプール

生成日：YYYY-MM-DD
原資：articles/published/ 全記事

| No | 来週使用 | 出典記事 | 型 | FW | タイプ/タイプ言及なし | 金言フレーズ（核） | 一言設計メモ |
|---|---|---|---|---|---|---|---|
| 1 | ✅ | ①既読スルー... | 型α | MBTI | ISFJ/FCPE | 「〇〇じゃなくて△△だっただけ」 | ... |
| 2 |  | ... | | | | | |
（10行）
```

**「来週使用」列のチェック基準**
- `sns_post_cheatsheet.md` の基本設定にあるFW比率（数値はそちらが唯一の正）を満たす組み合わせを優先的に✅にする
- 週1本必須の型α・週1本試験枠の二つの温度型が含まれるよう✅を選ぶ
- 議論誘発型・攻撃的ラベリング型は✅にしない（そもそもStep1で除外済みのはず）
- このプールは10件のアイデアバンクであり、週の必要本数（現行は週21本×2媒体）を1回の実行で満たす必要はない。X・Threads分は別途 `/mbticode-post` で角度を変えて本文化する。プールを使い切ったら再実行する

---

## Step 4：出力後の確認

- [ ] 10件すべてに出典記事（articles/published/のファイル名）が明記されているか
- [ ] FW比率・型の重複が偏っていないか（同じ型が10件中5件以上を占めていないか）
- [ ] 有料記事(②③)発の種が、テンプレの丸写しになっていないか
- [ ] 直近の `posts_x.txt` / `posts_threads.txt` と重複していないか
- [ ] 10件の金言フレーズすべてに `/quality-guardrail` を通し、検出0件を確認したか（〜ですね／〜と思います／〜が重要です等のAI的な薄い表現が残っていないか）

完了したら「アイデアプール◯件生成・来週使用候補◯件を scheduled_posts.md に保存（quality-guardrail済み）」の1行で報告する。

---

## 本文化する場合

ユーザーが特定のNoを選んで本文化を依頼したら、`sns_post_cheatsheet.md` の生成ワークフロー（Step 2以降・発信前チェックリスト）を適用して1本ずつ仕上げる。10件を一括で本文化しない。
