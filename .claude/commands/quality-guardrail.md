# Quality Guardrail（AIっぽさ添削）

SNS投稿・CTA自己リプライの生成後、保存前に必ず通す最終添削パス。
`sns_post_cheatsheet.md` のブランド文体ルール（じゃない/じゃなくて・けど・てしまう・禁止記号等）を**置き換えるものではなく、その上に重ねる最終フィルター**。両方満たして初めて保存可能とする。

## 呼び出し方

```
/quality-guardrail [対象テキストまたはファイルパス]
```

`/mbticode-post` `/mbticode-content-pipeline` の内部ステップとしても自動的に呼び出される（後述「既存フローへの組み込み」参照）。単体でも、既存の`posts_x.txt` / `posts_threads.txt` / `cta_templates.md` の添削し直しにも使える。

---

## このSkillの役割

「文法的には正しいが、AIが書いたと分かる薄い表現」を検出し、`persona_core.md` の期待値（責められない分析・解像度が上がる感覚・もう少し先が知りたい余白）に沿った短く断定的な文体へ書き直す。

`mbticode_tone.md` / `sns_post_cheatsheet.md` が禁止しているのは「記号・語尾レベル」の問題（`""`・`──`・ではない等）。このSkillが狙うのはそれとは別の層、**構文レベルの薄さ**（ヘッジ・説教口調・当たり障りのない相槌）。

## Step 1：禁止表現リストと照合する

以下は原則すべて禁止。見つけたら該当箇所を必ず書き直す（削除だけで終わらせない＝薄くなった分を断定文で埋め直す）。

### 相槌・ぼかし語尾

| 禁止 | 理由 | 代替の方向性 |
|---|---|---|
| 〜ですね | 同意を求める相槌。断定を避けている | 言い切る。「〜だ」「〜んです」（cheatsheetの回数制限内で） |
| 〜と思います / 〜と感じます | 自信のなさの表明。設計官の語り口と矛盾 | 観察として言い切る。「〜んです」「〜だった」 |
| 〜でしょう / 〜ではないでしょうか | 遠回しな問いかけで逃げている | 直接の問いにするか、言い切りにする |
| 〜かもしれません（多用） | 予防線。1投稿に何個も入れると説教くさくなる | 具体的な条件文に置き換える（「〜なら」「〜のとき」） |
| 〜という感じです | 抽象化して逃げている | 具体的な行動・シーンで言い切る |

### 提案・説教口調

| 禁止 | 理由 | 代替の方向性 |
|---|---|---|
| 〜しましょう / 〜していきましょう | セミナー講師口調。内声モードと相反する | 命令ではなく観察を差し出す。CTAで行動を促す場合のみ「〜してみてください」に留める（cheatsheetのCTAルール準拠） |
| 〜することが大切です / 〜が重要です | 抽象的な総括で終わる典型的なAI文末 | 「重要」を名指しせず、なぜそうなるかの構造で語らせる |
| 〜することをお勧めします | コンサル口調。アカウントのポジション（先輩・設計官）と合わない | 「〜した方がいい」も避け、体験・観察から自然に導く |
| 以上のように / このように / まとめると | 説明の要約フレーズ。SNS投稿では文章が長く見える上に機械的 | 削除し、直前の内容で終わらせるか金言フレーズで締める |

### 抽象名詞の多用

| 禁止 | 理由 | 代替の方向性 |
|---|---|---|
| 〜性（関係性・必要性・重要性等）の連発 | 抽象名詞を積み上げると内容が薄く見える | 具体的な行動・セリフに置き換える |
| 個人差があります / 一概には言えません | 責任回避の予防線。設計官としての言い切りと矛盾 | 削除する。言い切った上でN（絞り込み）文で対象を絞る方が誠実 |

## Step 2：persona_core.mdに基づく文体強制

書き直す際は必ず以下を満たす。

- **短く**：1文を短く区切る。「〜で、〜で、〜だから」のような複文の連結を避ける
- **断定的**：観察や構造の話として言い切る。「〜んです」「〜だ」「〜てしまう」はcheatsheetの回数制限（なんです系1投稿1回・だ多用禁止）を超えない範囲で使う
- **責めない**：断定はするが、読者を否定しない。「あなたが悪い」ではなく「設計がそう動いた」という構造の言い切りにする
- **余白を残す**：全部を説明し切らない。未解決設計の原則（なぜか／どうすればか、どちらか一方を書かない）を保つ

---

## Step 2.5：同日投稿間の角度重複チェック（1日複数本投稿する場合・必須）

2026-07-05notekaigiでX本文投稿が1日3本に増量されたことに伴い追加。同じ日に複数本投稿する場合、以下を確認する。

- 同日の他の投稿と「AとBは別の話」型の区別・逆転の軸が同じになっていないか
- 同じFW（同じMBTIタイプ・同じラブタイプキャラクター）を同日に重複させていないか
- 同じ型（型α・型B等）を同日3本すべてに使っていないか（最低1本は型を変える）
- 3本を続けて読んだときに「同じことを3回言っている」という印象にならないか

検出した場合、後の投稿の角度を別のタイプ・別の型に差し替える。差し替え候補がなければ、その日の本数を減らすことを優先する（無理に3本目を薄い内容で埋めない）。

## Step 3：添削結果の出力

```
【検出】〇件（禁止表現／該当箇所）
1. 「〜だと思います」→「〜んです」に修正
2. 「〜が重要です」→ 削除し、直前の観察で言い切りに変更
...

【添削後】
（修正済み本文全体を表示）

【確認】
- [ ] cheatsheet側の文体ルール（じゃない/じゃなくて・けど・禁止記号）と衝突していないか
- [ ] 断定にしすぎて「読者を責める」文になっていないか
- [ ] 未解決設計の原則（どちらか一方を空白にする）が保たれているか
```

検出0件の場合も「検出0件・そのまま保存可」と1行で報告する。

---

## 既存フローへの組み込み

- `/mbticode-post` Step 3（Threads本文・X投稿の内部生成後、`posts_threads.txt` / `posts_x.txt` へ保存する前）に本Skillの添削パスを必須ステップとして挿入する
- `/mbticode-content-pipeline` でアイデアの金言フレーズを書き起こす際、および`cta_templates.md`に新規CTAパターンを追記する際も、保存前に本Skillを通す
- 発信前チェックリスト（cheatsheet）とquality-guardrailは**両方通過して初めて保存可能**。どちらか一方を省略しない
