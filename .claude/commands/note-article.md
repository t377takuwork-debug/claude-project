# Note記事生成システム

Note有料・無料記事をテーマ入力から完成まで一貫して生成するシステム。
会議は行わない。確定済みの原則を直接適用し、ユーザーの承認ポイントのみで進む。

## 呼び出し方

```
/note-article [アカウント名] [テーマ/ネタ]
```

例：
- `/note-article mbticode 繰り返す恋愛パターン`
- `/note-article s4lv_ai Claude Codeで記事設計を効率化する`

## テーマ未指定時の処理

アカウント名のみ指定でテーマが未指定の場合：
- **mbticode**：`mbticode_strategy.md`・`mbticode_tasks.md`・公開済み記事インデックスを参照し、現フェーズと既存記事との重複を避けた上でテーマ候補を3案提案する。各案に「ターゲット直撃度・フラッグシップへの導線・差別化」の所見を添える
- **その他**：「アカウント名とテーマを教えてください」と一言だけ聞く

## Step 1：ペルソナ設計

「たった一人」の人物を描く。書き手の羅針盤として使う（タイトルには出さない）。

**出力する項目**
- 年齢・属性・職業・状況
- この人が夜な夜な実際に検索する言葉（3〜5個）
- 今感じている感情（自己嫌悪・不安・怒り・疑問など具体的に）
- 求めている変化
- この人がお金を払う瞬間

**ユーザー承認ポイント①**
「このペルソナで進めますか？」と確認する。

---

## Step 2：タイトル生成

確定したペルソナを起点に、タイトル案を3つ生成する。

**ルール（変更不可）**
- 感情・行動・状況で書く（タイプ名・フレームワーク名をタイトルに入れない）
- Note編集部の3条件：具体性・読者メリットの明確さ・差別化
- ケープルズの4特徴：得になる・新情報・好奇心・手っ取り早さ
- サブタイトルは任意。付ける場合は「手に入るもの」を明示する
- Noteタイトル欄は45字以内を目安にする

**チェックリスト（3案すべてに適用）**
- [ ] 読者の「得になる」要素があるか
- [ ] タイプ名・フレームワーク名を含んでいないか
- [ ] 暗い・否定的なイメージを植え付けていないか
- [ ] 一目で意味がわかるか
- [ ] 競合にない差別化された切り口か

**出力フォーマット**
```
案A：[タイトル]
案B：[タイトル]
案C：[タイトル]

| 条件 | 案A | 案B | 案C |
|---|---|---|---|
| 読者の得 | | | |
| 手に入るものの明確さ | | | |
| 差別化 | | | |
| 感情フック | | | |
```

**ユーザー承認ポイント②**
「どの案を選びますか？または修正指示をください」と確認する。

---

## Step 2.5：SEOキーワード確認（毎回必須）

タイトル案の承認前に、ペルソナが実際に検索する語句をWebSearchで確認する。

**目的**
Note記事はSNS流入だけでなく検索流入も重要な経路となる。ペルソナが「夜に感情的に打ち込む言葉」がそのまま検索語句になるため、タイトルにそれを組み込むことでSEOと感情フックを同時に狙う。

**実行手順**

① Step 1で特定した「夜な夜な検索する言葉」を3〜5語句、WebSearchで実在性を確認する
② 検索結果から以下を判断する：
   - 実在するキーワードか（競合記事が存在するか）
   - 競合の切り口はどんな軸か（カウンセリング系・情報系・体験談系など）
   - MBTICODEの差別化余地はあるか（「設計・構造・MBTI×ラブタイプ」軸）
③ 採用するキーワードを決定し、Step 2のタイトル案を「感情系キーワードを含む形」に修正して最終3案を出力する

**キーワード採用基準**
- 感情語句であること（「合わせすぎる」「自分が消える」「尽くしすぎる」等）
- 娯楽系キーワード（「診断 面白い」「タイプ当てクイズ」等）は除外
- タイプ名・FW名はタイトルに含めない（既存ルール維持）

**流入元の判断**
- SNS流入が安定している場合：感情フック優先、キーワードは自然に入る範囲で
- SNS流入が弱い・フォロワーが少ない場合：感情系キーワードをタイトルに必ず組み込む

## Step 3：見出し構成設計

**3視点の役割分担**

| セクション | 担当視点 | 内容 |
|---|---|---|
| 導入部 | 認知ハック・人間心理 | ターゲットの最大の痛み＋ベネフィット＋フック |
| 本論 | 論理的逆算・職人品質 | H2：3〜6本、各H2に執筆方針を付ける |
| 結論 | コミュニティ・物語設計 | 1つのアクション＋あえての隙 |

**H2の設計ルール**
- 読了率を最大化する本数に絞る（一律固定しない）
- 各H2に「何を・どこまで具体的に書くか」の執筆方針を必ず付ける
- ニッチ義務化：検索結果の要約記事禁止
- **H2タイトルにStep 2.5で確認した検索キーワードを自然に組み込む**

**H3の設計ルール**
- 各H2について「H3が必要かどうか」を必ず検討する
- H3を使う基準：H2の内容が複数の独立したサブトピックに分かれる場合・段階・パターンを列挙する場合
- H3タイトルにも可能な範囲でキーワードを組み込む（追加の検索流入設置点として機能させる）
- H3は1つのH2に2〜3本が上限（多すぎると読了率が下がる）

**出力フォーマット**
```
## 導入部
- ターゲットの痛み：
- ベネフィット：
- フック：

## 本論
H2①：[見出しタイトル（キーワード含む）]
　執筆方針：
　H3：[サブ見出し]（不要な場合は省略）
　H3：[サブ見出し]

H2②：[見出しタイトル（キーワード含む）]
　執筆方針：
　H3：（必要な場合のみ）
（必要な数だけ展開）

## 結論
- N（絞り込み）：
- 1つのアクション（CTA）：
- あえての隙・問いかけ：
```

**ユーザー承認ポイント③**
「この構成で進めますか？」と確認する。

---

## Step 4：記事本文生成

承認された構成に従い、本文を生成する。

**適用するルール**
- `writing_core.md` のPASONA・5大ルール・親近感手法を適用する
- `writing_expressions.md` から状況に応じたパターンを選択する
- `CLAUDE.md` の絶対遵守ルール（Note規約・捏造禁止）を守る
- mbticodeは `mbticode_tone.md` の全ルールを適用する（文体・記号・記事構成定型）

**mbticode 無料記事の品質基準**

| 基準 | 内容 |
|---|---|
| 最低文字数 | 2,500字以上（目標2,500〜3,000字） |
| 場面描写 | タイプ名の羅列禁止。「こういうとき、こう動いてしまう」という行動レベルのシーンを必ず入れる |
| 持ち帰れるもの | テンプレート・即実践できる問いかけを最低2つ以上 |
| 有料との差別化 | 「構造の理解」を無料で渡す。「タイプ別の具体設計図・出口」は有料に残す |

**mbticode 視覚フォーマット（有料・無料共通）**

| 要素 | 使用箇所 | 禁止 |
|---|---|---|
| `>` blockquote | 読者の心の声・設計の核心フレーズ | 通常の説明文・自己開示 |
| ` ``` ` コードブロック | 読者が使えるテンプレート例文 | 地の文 |
| `**` 太字 | 設計の核心フレーズのみ | 自己開示・説明文 |
| `###` H3 | H2の内容をさらに深掘りする場合 | H2と同列に並べる |
| `---` | セクション区切り | 多用しない |

---

## Step 4 完了後：公開前チェックリスト（mbticode）

本文を生成したら、出力前に**本文の該当箇所を実際に目視確認しながら**全項目を採点する。
「読んだ」だけで通過させない。各項目は本文と照合して判定すること。

**品質チェック**
- [ ] 2,500字以上あるか
- [ ] 行動レベルの場面描写が入っているか
- [ ] 持ち帰れるテンプレート・問いかけが2つ以上あるか

**トーン・記号チェック（`mbticode_tone.md` 照合）**
- [ ] 読者の心の声に `>` blockquote を使っているか
- [ ] 設計の核心フレーズに `>` blockquote または `**` 太字を使っているか
- [ ] テンプレート・例文がある場合 ` ``` ` で囲んでいるか
- [ ] `""` ダブルクォートが残っていないか（本文全体を検索して確認）
- [ ] `──` ダッシュが残っていないか（見出し含む全体を確認）
- [ ] `ではない/ではなく` が残っていないか（本文全体を検索して確認）
- [ ] 自己開示に一発逆転ストーリーが完結しているか
- [ ] N（絞り込み）が結論前に入っているか
- [ ] CTAが1つで時間軸付きか・記事内番号を参照していないか

**末尾の固定要素チェック**
- [ ] 有料記事への誘導文があるか（`article_index.md` の実在URLを使用）
- [ ] 有料記事リンクの前後を含め、価格表記が一切ないか（リンクテキスト・リンク直後・同一行すべて確認）
- [ ] 発信者フッターがあるか（`恋愛の設計局｜繰り返すパターンの出口を見つける` / `X：[@MBTICODE](https://x.com/MBTICODE)`）
- [ ] 管理情報欄にNoteタグが記載されているか（標準タグ：`ラブタイプ` `MBTI` `恋愛` `繰り返す恋愛` `恋愛パターン`）

## Step 5：ドラフト保存（必須・本文出力と同時に実行）

チェックリスト全項目通過後、`mbticode/articles/drafts/` にファイルを保存する。

**ファイル名**：記事テーマを簡潔に表した日本語（例：`合わせすぎて自分がわからなくなる理由.md`）

**ファイル先頭に必ず記載する管理情報**：
```
作成日：YYYY-MM-DD
種別：無料記事 or 有料記事（価格）
有料誘導先：①〜⑥の番号（URL）
Noteタグ：タグ1 / タグ2 / タグ3 / ...
```

**保存のタイミング**：本文をユーザーに出力するのと同じターンで保存する。「後で保存する」は禁止。

## 全体の流れ

```
[テーマ未指定の場合] テーマ3案提案 → ✅選択
Step 1：ペルソナ設計 → ✅承認
Step 2：タイトル生成（3案）→ ✅選択前に↓
Step 2.5：SEOキーワード確認（WebSearch）→ タイトル最終案に反映
✅タイトル承認
Step 3：見出し構成設計（H2・H3・キーワード配置）→ ✅承認
Step 4：記事本文生成 → 公開前チェックリスト全通過（本文と目視照合）
Step 5：drafts/ に保存（本文出力と同ターンで実行）→ 完成
[任意] 公開前品質チェック → /notekaigi を呼び出す
```

## Note投稿時の運用（mbticode）

- **サブタイトルは任意**：記事ごとに必要かどうか判断する
- **Noteタイトル欄**：45字以内を目安。超える場合はメインタイトルのみ入力する
- **Noteタグ**：ドラフトの管理情報欄に記載したタグを設定する

---

## 注意事項

- 各ステップで確定済みの原則は再議論しない
- 戦略的判断が必要な場面（新アカウント設定・方針転換・想定外のテーマ）は `/notekaigi` を別途呼び出す
- アカウント情報がない場合はStep 1の前にユーザーへ確認する
