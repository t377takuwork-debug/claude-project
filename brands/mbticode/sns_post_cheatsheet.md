# ▼ 生成ワークフロー（毎回この順序で実行）

**Step 1. 参照ファイル確認**（タイプ言及がある場合のみ）
- MBTI → `reference/mbti_quick_ref.md`
- ラブタイプ → `reference/lovetype_sns_ref.txt`（詳細必要時のみ `reference/ラブタイプ各キャラクター傾向まとめ.txt`）
- DSKB → `reference/dskb_quick_ref.md`

**Step 2. X投稿9本 ＋ Threads設計テーブルを同時提示**
- X投稿9本を完成品として提示する
- Threads事前設計テーブルも同時に提示してユーザーに確認を求める

**Step 3. ユーザーのOK後にThreads本文9本を一括生成**

**Step 4. 発信前チェックリストで自己採点（全項目通過後に出力）**

**Step 5. `brands/mbticode/posts/posts.txt` に上書き保存して完了**
- ファイル名：固定（`posts.txt`）。毎回上書きする
- 保存順：X投稿9本 → Threads投稿9本 → 運用メモ
- 行頭スペースなし・文中の不自然な改行なし（コピペ即使用できる状態）

# ▼ 発信前チェックリスト（出力前に全項目確認）

**X投稿**
- [ ] 最初の2〜3行で「何を言っているか」が分かる
- [ ] 金言フレーズが1〜3行目に置かれている
- [ ] タイプ名は金言フレーズの後（中盤以降）にある
- [ ] Threadsの圧縮版になっていない（別コンテンツとして設計されている）

**Threads投稿**
- [ ] 9本中5本以上に体験の匂わせが入っている
- [ ] タイプ名は本文中盤以降にある
- [ ] 完結型5本・続き型2本・URL事後型2本の割り当てが守られている

**全投稿共通**
- [ ] 全投稿に「設計」ワードが入っている
- [ ] 禁止語（ではない／ではなく／でも接続詞）がない
- [ ] 禁止記号（""・──）がない
- [ ] 体言止め連打がない
- [ ] 「なぜそうなるか」か「どうすれば変えられるか」のどちらか一方が意図的に空白になっているか（両方書くと消費されて保存されない）

**週単位**（9本一括生成時のみ適用）
- [ ] 逆転構造が週1本以上ある
- [ ] FW比率（MBTI3／ラブタイプ3／なし2／DSKB1）を満たしている
- [ ] 出口・発見系が週2本以上ある
- [ ] 痛み・繰り返し系が週5本以内に収まっている
- [ ] 締め前フレーズが金言テスト（逆説・対比・構造命名・自分への問い・命名）を通過している

# ▼ 共通ルール

## 基本設定

- **投稿時間**：08:00 / 12:00 / 22:00（X・Threads共通）
- **ハッシュタグ**：なし
- **FW比率（週9本）**：MBTI 3本 ／ ラブタイプ 3本 ／ タイプ言及なし 2本 ／ DSKB 1本（月2本上限）
  - DSKB月2本を使い切った週は DSKB枠を「タイプ言及なし」に振る（MBTI3／ラブタイプ3／なし3）
- **内容の根拠**：`reference/` のみ（架空データ禁止）

## 時間帯と型

| 時間 | 型 | 読者の状態 |
|---|---|---|
| 08:00 | 型B（解釈転換） | 通勤中・視点を変える気づき |
| 12:00 | 型A（シーン共感） | 昼休み・流し見でも止まる |
| 22:00 | 型A または 型C | 就寝前の内省 |

## 4種の型

**型A（シーン共感）**
行動シーンで共感 → 構造解説 → タイプ名に着地 → 他者投影
- 他者投影：「心当たりある人の顔が浮かびませんか？」「これ、周りの誰かに当てはまりませんか？」（明示的なシェア促しは禁止）
- ラブタイプ投稿：キャラクター名で共感 → 行動シーンで展開 → アルファベットで固有化 → 圧縮フレーズ

**型B（解釈転換）**
冒頭は下記4パターンから選択。週3本で同じパターン禁止。
- 観察型：「〇〇な人を見ていると、ある共通点がある。」
- 問いかけ型：「なぜ〇〇してしまうのか、ずっと気になっていた。」
- 独り言型：「これ、ずっと言いたかった話です。」
- 従来型：「〜という経験がある人へ。」（週1本まで）

**型C（問い＋自答）**
問いを立てて自分で答える。「どう思いますか？」で終わらせない。

**型D（DSKB・エンタメ×分析）**
設計官がDSKBで人間の行動を面白く解析する。月2本。Threadsに向いている。
- 「ドスケベ」文脈には乗らない
- 締め定型：「どれかひとつでも当てはまる人は、設計の話をしましょう。」
- 停止基準：4本連続で保存数が通常投稿の2倍未満 → 一時停止

## ネタローテーション

週9本：痛み・繰り返し系 最大5本 ／ 出口・発見系 最低2本

### 痛み・繰り返し系（最大5本）
| カテゴリ | シーン例 |
|---|---|
| 別れ後 | SNS確認・連絡しようとしてやめる |
| 関係中 | 言いたいことが言えない・尽くしすぎる・合わせすぎる |
| 選択 | 付き合えそうな人を遠ざける・好きじゃない人に安心する |
| 診断後（問題） | 調べたのに変わらない |
| 繰り返し | また同じタイプ・別れた理由が前回と同じ |

### 出口・発見系（最低2本）
| カテゴリ | シーン例 |
|---|---|
| 設計が変わった瞬間 | 「ああ、これか」と気づいた体験 |
| タイプを知る面白さ | 診断名がついた瞬間の「しっくり感」 |
| 相性の化学反応 | 特定タイプの組み合わせ・なぜ惹かれるか |
| 診断後（出口） | 設計図が手に入った感覚 |
| 許可を与える | 「〇〇でいい」型の解放 |
| 感情の分解 | 「〇〇と△△は別の話」 |

使用済みネタ確認 → セッションメモリの「投稿済みネタ記録テーブル」（project_sns_design.md）

### 採用しない型（高インプレッション・低保存率パターン）

保存率0.1%未満の型はMBTICODEのマネタイズ動線と切れているため、インプレッションが高くても採用しない。

| 型 | 特徴 | 排除理由 |
|---|---|---|
| 攻撃的ラベリング型 | 「異常者」「まともじゃない人」等のネガティブ命名 | 文体設計違反・炎上リスク |
| 議論誘発型 | 「〇〇って偏見だよね」系の反論を呼ぶ構造 | リプライ多・保存率極低・転換率ゼロ |
| エンタメ消費型 | DSKBの笑い文脈・タイプ比較ネタ | 保存されず有料記事への動線にならない |

## トーン・文体ルール

| 使う | 禁止 |
|---|---|
| じゃない / じゃなくて | ではない / ではなく |
| けど | でも（接続詞） |
| 〜てしまう | 体言止め連打 |
| 〜んです / 〜なんです | 〜だ（多用禁止） |

禁止記号：`""` ダブルクォート　／　`──` ダッシュ

- 読点の間引き：「して、やめる」→「してやめる」
- 疑問文は疑問符
- 空行でリズム

中学生上位レベルを基準（平易に・幼稚にしない）。

**「設計」**：全投稿維持必須。バリエーション：`設計が動く` / `設計を変える` / `設計ミス` / `設計図` / `設計がズレた`（禁止：`設計が狂っていた`）

**「回路」**：不可抗力感の文脈のみ。1投稿1回まで。それ以外は `パターン` / `癖` / `動き方` に置き換え。

**技術語の連続禁止**：`処理` / `コスト` / `組み込まれている` / `機能していない` は1文に2語以上連続禁止

差し替え辞書：
| 元の表現 | 修正後 |
|---|---|
| コストとして処理する回路が組み込まれている | 自動で避けてしまう癖がある |
| 傷つく前に終わらせる処理が先に走って | 傷つく前に自分で終わらせてしまうパターンがある |

## 逆転・反転構造（週1本以上必須）

高反響パターン。X金言フレーズに最優先で使う。

| 型 | 例 |
|---|---|
| 「〇〇じゃなくて△△だっただけ」 | 「未練じゃなくて設計に名前がついていない状態だっただけ」 |
| 「〇〇してるんじゃなくて△△してる」 | 「聞いてないんじゃなくて、聞きすぎてる」 |
| 「〇〇したくないんじゃなくて△△したいだけ」 | 「仲良くしたくないんじゃなくて役割として分けたいだけ」 |

シーンの具体度：「連絡しようとしてやめる」→「文章を打って送らずに消す」レベルまで落とす。

## ブックマーク誘発設計（金言テスト）

締め前の圧縮フレーズに「この一文だけスクショしたくなるか？」を自問する。
- 合格基準：逆説・対比 ／ 構造に名前をつける ／ 自分への問いになる ／ **抽象的な感覚に固有名詞を与えている**
- 不合格例：「設計を変えることで関係の見え方が変わります」（説明的）
- 合格例：「連絡できなかったんじゃなくて、設計が先に手を止めてしまってた。」
- 命名合格例：「情緒のサービス料がゼロな関係」「設計の終着点」「手放せない構造」

## 未解決設計の原則

保存は「また見返したい」ときに起きる。「わかった」で終わると消費される。

- 投稿内で「なぜそうなるか」か「どうすれば変えられるか」のどちらか一方を意図的に書かない
- 両方書いた投稿は共感で消費されいいねは増えるが保存されない
- 空白にした問いが有料記事への自然な橋渡しになる
- 「手元に置いて考えたい」と思わせることが目的

## 改善トリガー

| 症状 | 対応 |
|---|---|
| 22時投稿の反応が薄い | 型Cの自答部分を強くする |
| 08時投稿が伸びない | 型Bの解釈転換の切り口を変える |
| Noteフォロワーが増えない | 自己リプライの補足文を「無料で読める」に変更 |
| 保存数が少ない | 金言テストの基準を上げる・締め前フレーズを書き直す |
| いいねは多いが保存が少ない | 「両方書きすぎ」が原因。未解決設計の原則を確認し、どちらかの問いを削る |
| インプレッションは高いが転換しない | 採用しない型に近いコンテンツが混入している。ネタローテーションを点検する |

# ▼ X専用ルール

## 制約

- **文字数**：140字以内
- **URL**：本文に入れない
- **ラブタイプのアルファベット**（FCPE等）：140字の制約上、自己リプライに回してもよい

## 役割と構造

**X = 「扉を叩く」型。Threadsの圧縮版にしない。**

金言フレーズ1本で完結させる。構造展開は捨て、逆転フレーズ・命名フレーズを核にした別コンテンツとして設計する。

推奨フォーマット：
```
[シーン or 問い：1〜2行]

[逆転または命名フレーズ：1行 ← ここが全体の核]

[余韻 or 補足：1〜2行（なくてもよい）]
```

- 核心フレーズは必ず1〜3行目に置く
- タイプ名は核心フレーズの後に補足として置く（書き出しに置かない）
- 「〇〇の話です。」型の中間説明行は削除または末尾移動

## 自己リプライ（X）

URL誘導型のみ。反応が来た投稿に後から追加（新規記事公開日のみ投稿直後に追加可）。
内容：1〜2行の誘導文 ＋ URL

# ▼ Threads専用ルール

## 制約

- **文字数**：200〜350字
- **URL**：新規記事公開日のみ本文末尾可。それ以外は自己リプライ

## 役割と構造

**Threads = 「中に入る」型。なぜそうなるかの構造展開 + 体験匂わせを必ず入れる。**

**体験匂わせ（週5本以上必須）**：
- 位置：冒頭または本文2〜3行目
- 例：「自分もそうだったので」「同じことをやってきたから言えるけど」「これを見抜けなかった時期があって」

**タイプ名は中盤以降**：書き出しにタイプ名を置くと「自分のタイプじゃない→スクロール」が発生する。シーン共感で書き出し → 中盤でタイプ名に着地。

## 事前設計テーブル（X投稿と同時に提示）

| 曜日・時間 | 型 | FW・タイプ | テーマ系統 | リプライ役割 |
|---|---|---|---|---|
| xx08 | 型B | MBTI（INFP） | 出口系 | 完結 |
| xx12 | 型A | ラブタイプ（FCPE） | 繰り返し | URL事後型 |
| xx22 | 型C | なし | 出口系 | 続き型 |
| ...以下9行 | | | | |

ユーザーがOKを出したら本文9本を一括生成する。

## 自己リプライ週次割り当て（Threads）

週9本を3役割に割り当てる：

| 役割 | 本数 | 構造 |
|---|---|---|
| 完結型 | 5本 | 本文だけで完結。リプライなし |
| 続き型 | 2本 | 本文に「続きがある」構造を仕込み、リプライを最大4段まで展開。URLなし |
| URL事後型 | 2本 | 完結型の中で反応が大きかったものに後からURLリプライ |

- コンテンツ系リプライ：80〜150字
- URL誘導リプライ：1〜2行＋URLで完結（水増し禁止）

続き型→URL誘導への切り替えは積極的に行う（記事⑤が最も使いやすい）。
